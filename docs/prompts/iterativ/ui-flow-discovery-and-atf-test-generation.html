2. Flow-uri si diagramebanda unui singur run UI
2.1 Banda UI pentru un singur runlibertate tactica, audit strict
Cum citesti banda
  • Seed este doar intrarea, nu verdictul.
  • Descopera UI inseamna rute, stari si intentii vizibile, nu replay de recorder.
  • Alege frontiera inseamna cea mai valoroasa mutatie business vizibila local.
  • Implementeaza felia trebuie sa lase o delta mica, dar countable.
  • Dovada + feedback trebuie sa poata fi recitite rece de orchestration si reporting.
2.2 Frontier UI: owned / partial / blockedlectura specifica adaptorului UI
Legenda
  • owned cere rezultat vizibil stabil + diff + proof.
  • partial poate repara suportul fara sa castige identitate business noua.
  • blocked inseamna ca UI-ul nu inchide rece frontiera, nu doar ca testul a fost incomod.
  • Daca un flow UI spune mai mult despre locator sau helper decat despre actor, intentie si rezultat, inca nu este frontiera buna.
2.3 Ce preda adaptorul UIpredare spre orchestration
Legenda
  • Adaptorul UI preda materia rece: claim, proof, diff, blockers si lectii.
  • Orchestration decide forma pachetului.
  • Core decide legalitatea.
  • Reporting decide pagina finala.

In raportul final, Business Flow Graph poate deveni diagrama, lane-uri scrise sau subgrafuri. Adaptorul UI trebuie sa predea catre orchestration materia rece a run-ului: actor, intentie, pas vizibil, rezultat owned, rezultat blocked si lectia locala decisiva. Nodurile trebuie sa ramana scurte si sa refoloseasca stilul comun din shared-report-reference.css.
Cand subgrafurile devin dense, asteptarea implicita este: legenda scurta, putin mai explicativa decat nodurile, colapsata imediat sub subgraf. Sectiunile UI secundare pot fi si ele colapsate implicit daca altfel pagina devine grea.
