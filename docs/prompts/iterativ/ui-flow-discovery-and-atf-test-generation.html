2) Discovery And Implementation Rules

2.1 Business-First Discovery DISCOVERY

Primary questions

  • What capabilities, journeys, actor states, and transitions does the application expose?
  • Which flows are covered, partial, deferred, blocked, or still undiscovered?

Required behavior

  • MUST inspect SEED_RECORDING_CLASS first.
  • MUST open and analyze the current local CANONICAL_HTML, ANALYSIS_HISTORY_ROOT, and preserved run packages when they exist before deciding the next frontier; if the repo was intentionally reset and these artifacts are absent, record that absence explicitly and use benchmark packages plus live discovery as the starting memory.
  • MUST reopen the strongest relevant benchmark canonical HTML in a real browser when it is locally reachable, and SHOULD reopen the benchmark history, run summary, and iterative prompt snapshot that most directly explain how that benchmark earned trust.
  • MUST distinguish benchmark memory reopened for judgment from execution bootstrap provenance needed only to make the run executable. If generated support, scaffolding, or execution-layer code is rebuilt from an external snapshot, record that as bootstrap provenance rather than as the benchmark itself.
  • MUST inspect existing tests under UI_TEST_ROOT and reusable support under UI_SUPPORT_ROOT.
  • MUST prefer deeper continuation of the same business flow when a visible next step exists and the current truth is still only shell-level, route-level, or narrowly targeted.
  • MUST keep a seed-adjacent frontier ledger so every meaningful route, CTA, tab, or hyperlink touched directly by the seed ends as covered, blocked, duplicate, or explicitly deferred with a concrete reason.
  • MUST inspect live routes, menus, hyperlinks, forms, validations, protected areas, and state transitions whenever confidence is not high enough.
  • MUST attempt meaningful accessible internal links and classify skipped links as duplicate, blocked, external, deferred, or low-value.
  • MUST treat separate environments, examples, external docs, and third-party destinations as distinct variants, not baseline coverage.
  • MUST treat LIVING_ARCHITECTURE_DOC_PATH as implementation guidance only, never as behavioral truth.
  • MUST NOT let page objects or support packages become the product model.

2.2 Seed Conversion Discipline UI
  • MUST treat the seed as a recorder artifact and initial evidence set.
  • MUST decompose it into routes, forms, validations, actor transitions, and business goals.
  • MUST remove recorder noise and split the seed into focused business tests.
  • MUST generate resulting business tests under TARGET_TEST_PACKAGE by default.
  • MAY override the default only with concrete repo-fit evidence.
  • MUST NOT preserve giant replay tests, recorder naming, redundant clicks, or selector noise.
  • MUST document how meaningful iterations changed understanding from seed clue to ATF outcome.
2.3 ATF Ownership And Shared-Layer Hardening UI
  • MUST create or update business tests under UI_TEST_ROOT.
  • MUST create or update reusable support under UI_SUPPORT_ROOT.
  • MUST preserve Playwright, Selenium, Extent, Allure, screenshot, and lifecycle compatibility.
  • MUST fix shared weaknesses where they belong: waits, route helpers, navigation helpers, menu helpers, validations.
  • MUST harden deterministic session setup when user, cart, account, or comparable state can leak between tests.
  • MUST NOT invent a framework-v2 abstraction layer or duplicate reporting/screenshot systems.
  • MUST NOT solve a shared weakness by scattering the same workaround across many tests.
Use the smallest safe improvement, but prefer shared repairs once a weakness is clearly reusable.
Flow pressure rule: do not abandon a business flow after the first decent proof if the next business step is visible, important, and realistically reachable. Either deepen it or state exactly where it blocked.
2.3.1 Business-Oriented Test Organization UI
  • MUST organize generated tests around discovered business domains and journeys, not recorder order.
  • MUST use focused class names that communicate user intent, state, and capability.
  • MUST split positive, negative, boundary, and recovery coverage when combining them would hide intent.
  • MUST keep page objects, services, validators, and fixtures as support layers.
  • MUST avoid hardcoded domain examples unless the live application actually exposes those domains.
Good organization should let a new engineer infer the business journey from the test package and class names before reading the implementation.
2.4 Locator, Wait, And Determinism Rules UI

Preferred locator order

  1. data-test
  2. accessible label
  3. role
  4. placeholder
  5. stable visible text
  6. stable CSS
  7. XPath only as a last resort
  • MUST prefer business-visible waits over passive waiting.
  • MUST keep tests rerunnable and isolated.
  • MUST investigate whether instability belongs to the product, the ATF helper layer, or test design.
  • MUST NOT use sleeps, arbitrary retries, brittle nth-child selectors, or catch-and-ignore logic that hides flakiness.
2.5 Honest Contract Rules DISCOVERY
  • MUST automate only behavior the UI actually exposes.
  • MUST assert only outcomes, messages, states, and transitions that are truly observable.
  • If the UI exposes only the start of a flow, test the visible entry point honestly.
  • If a route settles correctly before a full load event, harden the route or wait helper instead of rejecting a valid flow.
  • If a discovered experience is rendered as an overlay, split-screen guide, or in-page workflow instead of a route change, assert the overlay-specific text, step state, and controls instead of forcing URL-based assertions.
  • If visible UI discovery leads to a route that is more stable to revisit directly in ATF, it is acceptable to validate that route directly after the initial discovery path has already been proven and documented.
  • MUST NOT overclaim invisible, unsupported, or speculative behavior.

4) Iteration Loop

4.1 Mandatory Loop QUALITY 
discovery -> business_flow_extraction -> invariant_detection -> coverage_analysis ->
implementation -> validation -> reassessment -> HTML update ->
milestone save decision -> score update -> next iteration
An iteration is complete only after implementation, validation, reassessment, HTML update, milestone decision, and score update.
4.2 No Premature Stop Rule QUALITY
  • MUST plan to execute all EXECUTION_ITERATION_COUNT.
  • MUST continue even if a narrow suite passes or the score crosses EXECUTION_OVERALL_QUALITY_TARGET early.
  • MUST use later iterations to deepen coverage, challenge assumptions, and revisit valuable deferred areas.
  • MUST use large budgets such as 50 to explore accessible hyperlink descendants, seed-adjacent continuations, and guarded state branches until they are either promoted, deferred, or blocked with evidence. Budget size alone is not learning.
  • MUST report after the round how many iterations were actually meaningful and what each meaningful checkpoint changed: business depth, evidence diversity, benchmark understanding, frontier rejection, or retained absorption.
  • MAY stop early only if the remaining work is blocked, unsafe, low-value, or would require architectural drift, and only after proving that no seed-adjacent frontier, meaningful accessible hyperlink, or benchmark-significant gap remains open.
  • MUST NOT use low-value alone as an early-stop justification while seed-adjacent frontiers or benchmark-significant gaps remain open.
4.3 Evidence Package Per Iteration QUALITY
  • MUST record what changed in the business model.
  • MUST record what changed in tests or shared support.
  • MUST record validation evidence and what it proved.
  • MUST preserve the generated Extent report for every kept iteration under validation/reports/ExtentReport_*.html whenever Extent reporting is supported by the repo. A kept run package that lacks its Extent artifact is audit-incomplete even if canonical HTML, summary text, and XML are present.
  • MUST record whether a milestone HTML was saved and why.
  • MUST record the next highest-value safe improvement candidate.
4.4 Coverage Prioritization QUALITY
  • MUST prioritize high-signal user journeys before shallow page pokes.
  • MUST prioritize stable observable contracts before speculative edge cases.
  • MUST prefer one honest product slice plus reusable hardening over many brittle shallow checks.
  • MUST derive coverage from discovered behavior, not assumptions about what a typical app should contain.
4.4.1 Coverage Dimensions QUALITY
  • Positive: stable happy paths that prove real business value.
  • Negative: visible validation, blocked transitions, rejected inputs, and guarded areas.
  • Boundary: deterministic limits or messages exposed by the UI.
  • Navigation/state: route transitions, CTAs, accessible links, overlays, actor transitions, and blocked states.
  • Regression: high-value journeys and reusable framework contracts discovered during the run.

5) Implementation Workflow

5.1 Convert The Seed Flow UI
  1. inspect the recorded actions
  2. identify the real business journey
  3. remove recorder noise
  4. identify pages, actions, outcomes, and reusable steps
  5. identify missing support or weak support in the ATF
  6. split the journey into focused tests
  7. place the resulting tests under TARGET_TEST_PACKAGE
5.2 Discover Around The Seed DISCOVERY
  1. flows directly implied by the seed
  2. adjacent flows sharing routes, menus, or actor transitions
  3. guarded or conditional behavior the application visibly exposes
  4. visible navigation, progressive disclosure, internal hyperlinks, and menu entries
  5. recovery, fallback, cancellation, retry, or alternative-path entry points when actually present
  6. detail, edit, search, filter, confirm, or submit flows only if they expose a clear observable contract
Treat hyperlink discovery as safe exhaustion: cover, defer with reason, mark blocked, or prove duplicate.
5.3 Test Data And Isolation UI
  • MUST generate unique or collision-safe data where uniqueness matters.
  • MUST use isolated, rerunnable data.
  • MUST NOT couple tests by execution order.
  • If the application does not support cleanup, design data to avoid collisions instead of hiding the problem.
5.4 Engine And Reporting Validation UI
  • MUST preserve Playwright, Selenium, Extent, and Allure compatibility when those paths are maintained by the repo.
  • MUST validate the narrowest useful engine and reporting path first, then broaden when confidence increases.
  • MUST NOT treat compile-only validation as sufficient.
5.5 Source-To-Artifact Traceability TRACE

Run package richness

  • MUST preserve a readable chain from SEED_RECORDING_CLASS to discovered behavior, generated test, support change, validation command, and HTML evidence.
  • MUST mark whether a test was seed-derived, live-discovered, regression-hardening-driven, or support-layer-hardening-driven.
  • MUST record when a test mainly protects framework behavior exposed during discovery.
  • MUST snapshot the exact discovery prompt and referenced standards used for the kept run package so later comparison can explain why this run produced stronger or weaker output.
  • MUST keep comparative source-lock provenance explicit whenever the round intentionally compares or borrows another family's standards: list exact external files used, exact external files excluded, whether WIP/temp/experimental files were excluded, and whether the run stayed execution-symmetric.
  • MUST keep execution-bootstrap provenance explicit whenever the run rebuilds generated support, services, pages, or tests from an external snapshot. State the exact snapshot once, classify it as bootstrap provenance, and do not let it masquerade as reopened business benchmark evidence.
  • MUST make every kept run package auditable from its own files: concise summary, prompt snapshot, validation evidence or honest no-validation note, and generated/support deltas rich enough for later comparison.
  • MUST update iteration-ledger.md for every attempted iteration, including attempts that produced no code change and no milestone.
  • MUST preserve a defect-discovery run separately when a later greener retained slice narrows or bypasses the unstable frontier instead of truly closing it.
  • MUST copy stable validation artifacts into the kept run package instead of leaving proof only in transient target/ folders.
  • MUST treat the per-run Extent artifact as required validation evidence when Extent is supported: copy it into validation/reports/ExtentReport_*.html for each kept run and downgrade package completeness if it is missing.
  • MUST NOT leave a generated test without a visible business reason or framework-hardening reason.

6) Validation And Close-Out

6.1 Functional Reassessment QUALITY
  • MUST reassess business understanding, route discovery, hyperlink discovery, actor-state discovery, invariant quality, seed conversion, coverage, maintainability, and generated ATF test-result quality.
  • MUST reassess positive, negative, boundary, navigation, state, recovery, and fallback coverage when the live UI exposes those dimensions.
  • MUST report reporting, screenshot, engine, and AI/MCP readiness only as confidence signals.
  • MUST keep pushing seed-adjacent frontiers before treating distant optional exploration as more important.
6.2 Maven Validation Expectations QUALITY

Validation and render honesty

  • MUST run the narrowest useful Maven validation first.
  • MUST broaden validation once narrow runs are green.
  • MUST report all Maven commands actually run.
  • MUST report command, scope, failure if any, root cause, fix applied, and final result.
  • MUST NOT hide failing runs that led to a useful fix.
  • MUST preserve the actual generated report artifact even when the repo writes Extent or Allure output to a framework-default location instead of the requested run path; copy the produced artifact into the run package and record the mismatch honestly.
  • MUST preserve Extent per kept iteration, not only for the final retained winner. If an intermediate kept run changed what the team knows, its own Extent artifact must remain reopenable from that run package.
6.3 Finalization Guard QUALITY
  • MUST NOT claim completion merely because code compiles, one suite passed, or the recorded flow replayed successfully.
  • Completion requires passing validation.
  • Completion also requires no remaining obvious safe iterative improvements in scope, or a clearly documented reason why they were not pursued.
  • MUST state clearly whether full iteration depth was completed and whether the score target was achieved.
6.4 Final Report And Scoring CORE
  • MUST state business discoveries, generated or updated ATF tests, shared support hardening, validation evidence, HTML artifacts updated, remaining partial/deferred/blocked areas, and the next safe iteration.
  • MUST be explicit about anything not verified and avoid claiming coverage beyond observed product behavior.
  • MUST state whether each kept checkpoint delivered business progress, reporting-only improvement, orchestration-only improvement, or no durable progress.
  • MUST state the strongest current Benchmark Gap, the strongest current Frontier Gap, and whether this round materially reduced either one.
  • MUST state which exact step of the active flow was closed more deeply, and which exact step remains blocked if the flow still did not close.
  • MUST state the deepest defended business chain reached in the run, in one explicit arrow-like sequence, and identify the kept run package that anchors that claim.
  • MUST state separately what the round explored broadly versus what it proved deeply, especially when a high execution budget produced many side journeys or retained hardening passes.
  • MUST state whether a broader retained-close actually absorbed the proof-origin truth, or whether the broader slice only reran, narrowed, bypassed, or failed around that truth.
  • MUST mark render verification as Partial or Blocked if local visual reopening fails, and preserve a screenshot or blocker note instead of pretending success.
  • MUST say whether any remaining open work is still seed-adjacent, benchmark-significant, or only distant optional exploration.
  • MUST say whether any bootstrap provenance changed only executability or whether it also changed business depth. If it changed only executability, keep that claim out of the business-winner reasoning.
