LangGraph Business Understanding Reporting Standard shared standard

Reusable reporting contract for the langgraph-business-understanding.html family. This file standardizes only the durable reporting shape: business-memory structure, rendered-diagram trust, milestone usefulness, ATF-confidence visibility, and cross-run promotion evidence. Product-local thresholds, paths, frontier choices, and execution logic must stay upstream.

business-first reporting ATF trust visibility rendered diagrams cross-run evidence single upstream target
1) Purpose and ownership
Goal: every kept analysis family must explain the product, show what the generated ATF package truly proves, preserve how understanding evolved, and justify why the promoted baseline deserves trust.
Anti-pattern: a thin status page with a few scores and sections but without rendered diagrams, frontier truth, cross-run judgment, or visible ATF-quality evidence.
This standard ownsCalling prompt keeps local
Canonical section shape, milestone contract, diagram/render rules, scoreboard dimensions, history usefulness, and promotion trust rules. Business scope, route frontier, local score targets, run roots, validation matrix, commands, package names, and environment-specific interpretation.
Intent: standardize the reporting contract, not the execution strategy.

Business Flow InventoryMUSTTrack covered, partial, deferred, blocked, variant, and excluded journeys cleanly.
Coverage OverlayMUSTKeep covered, partial, discovered-but-untested, blocked, and deferred states honest.

Cross-Run ComparisonMUSTPrevent latest-run bias and explain promotion by evidence.
HTML History IndexMUSTMake every milestone explorable and worth revisiting.

Deferred Frontier / Recommended Next IterationMUSTKeep future work concrete, bounded, and honest.

3) Milestone contract
Milestones must read like rich case notes. They are not numbered breadcrumbs and they are not trimmed copies of the canonical page.
FieldExpectation
Milestone themeOne real shift in business understanding, frontier truth, stability, trust, or prompt quality.
DeltaWhat changed since the previous kept state.
Observed pressureWhat weakness, ambiguity, or failure triggered the iteration.
Fix or decisionWhat changed in prompt, code, or interpretation.
Score snapshotAnalysis quality, generated ATF quality, prompt quality, and reporting overall quality at that milestone.
Next frontierWhat should be attempted next if the loop continues.
Hard failure: milestone families made of near-identical “still green” pages or snapshots that repeat the final winner without adding new learning.
4) Diagram and render contract
Preferred baseline: static inline SVG diagrams, because they do not rely on runtime Mermaid rendering and remain stable across local file open, simple static hosting, and artifact promotion.
  • Diagrams MUST render without external dependencies.
  • Diagrams MUST remain readable on local file open and simple static hosting.
  • If a source format is kept for editing, the rendered SVG still MUST be present in the final HTML.
  • Raw diagram source MUST NOT be the final user-facing output.
  • Every canonical HTML MUST include a readable business domain graph, actor-state graph, and flow graph.
  • Dense diagrams MUST be backed by ledgers or inventories that make them teachable.
  • Render verification is mandatory before promotion.
5) Scoreboard and trust rules
DimensionExpectation
business-first clarityThe artifact teaches the product before it teaches the framework.
canonical section completenessThe mandatory sections are present and useful, not nominal only.
milestone distinctnessThe history snapshots capture real shifts, not repetitive confirmations.
render and diagram qualityThe visual layer renders correctly and helps comprehension.
ATF trust linkageThe artifact explains clearly what the generated ATF package really proves.
cross-run explainabilityThe winner decision is explicit, comparable, and evidence-based.
history usefulnessThe milestone family is worth revisiting, not just archiving.
onboarding valueA new engineer can understand scope, trust, and next frontier quickly.
reporting overall qualityThe combined quality of the reporting package as a reusable business-understanding artifact.
ATF trust rule: the canonical HTML must contain the generated ATF quality score and the reasoning that supports it, not hide that truth only in meta review artifacts.
Single upstream parameter: this standard requires one upstream target only: REPORTING_OVERALL_QUALITY_TARGET. Numeric target, gating strength, and stop behavior belong to the governing prompt family, not to this shared standard.
6) Promotion and upstream integration
  • The canonical HTML MUST explain why the current run won when multiple governed runs exist.
  • The canonical HTML MUST keep deferred areas explicit and honest.
  • The canonical HTML MUST be rich enough to become a reference exemplar without requiring separate chat memory.
  • The execution prompt should reference this reporting contract instead of cloning it inline.
  • Only reusable reporting rules should live here; run-specific discovery, validation, and frontier logic should stay upstream.
  • The execution and meta prompts should point to the same canonical sections, milestone expectations, and trust rules.
Promotion bar: a canonical HTML can become a reference exemplar only if a new reader can understand scope, trust level, cross-run winner logic, generated ATF confidence, and remaining frontier without opening any additional artifact first.
