LangGraph Business Understanding Reporting Standard shared standard

Canonical reporting contract for the langgraph-business-understanding.html family. This file owns the durable analysis shape: sober dark presentation, compact graph language, proof mapping, nuanced coverage vocabulary, narrative history, hyperlink frontier treatment, validation artifact references, and benchmark-aware trust explanation.

0) Standard at a glance OPEN keep visible by default
What this standard owns Canonical section shape, proof mapping, diagram/render contract, milestone usefulness, benchmark/trust comparison, and reporting density rules. What the calling prompt keeps local Business scope, route frontier, local budgets, commands, validation depth, score thresholds, and repo-specific interpretation. Local exemplar root When present, docs/goldenCopies/html-exemplars-to-follow is the local exemplar library. Reports stored there are reusable reporting benchmarks, not just old winners. Why this page exists To keep reporting rules reusable and converged across ATF families instead of rediscovering them in each local run.
Best reading order: start with canonical contract, then presentation/diagram rules, then proof vocabulary, then milestone/history, and only then promotion details.
1) Purpose and ownership what this standard owns
Goal: every kept analysis family must explain the product, show what the generated ATF package truly proves, preserve how understanding evolved, and remain comparable across future quality-loop runs.
Anti-pattern: a page that looks polished but is still shallow: giant hero, oversized diagrams, table spam, weak coverage vocabulary, superficial hyperlink treatment, or benchmark comparison that says only "better / worse" without why.
This standard ownsCalling prompt keeps local
Canonical section shape, presentation contract, graph quality, proof-mapping structure, validation-reference rules, milestone usefulness, history form, run-lineage readability, and benchmark comparison rules. Business scope, route frontier, run roots, local scores, validation matrix values, commands, package names, and environment-specific interpretation.
Intent: good reporting rules must live here so future runs reproduce them automatically instead of relying on one locally improved artifact.
2) Canonical HTML contract required reusable section shape
2.1 Core verdict and identity first-screen truths
SectionStatusWhat it must accomplish
Executive SummaryMUSTFrame the product, current run identity, trust level, and remaining frontier in plain business language.
Trust stanceMUSTName the artifact honestly as a benchmark-grade winner, stronger local candidate, validated learning candidate, or non-promotable retained package. If a stronger benchmark still exists, say so plainly near the top.
Semantic Pill RailMUSTKeep a small top-level pill strip that carries useful truths such as primary run, baseline count, new promoted frontier, explicit non-promotion, main benchmark gap, and only the shortest useful provenance label when bootstrap or borrowed support mattered.
Business Summary / Trust SummaryMUSTSeparate what the product appears to be from what the current ATF package truly proves.
Current Baseline DecisionMUSTState which run is kept and why it displaced or failed to displace the prior best.
Primary Retained Validation RunMUSTName one primary kept run package that anchors the final verdict, and keep the wording aligned across Trust Summary, Validation Artifact References, and lineage sections.
Run Identity And Current ScoreboardMUSTShow run identity, stop decision, Business HTML score, ATF Package score, Overall score, target status, configured budget, and effective meaningful iterations early and clearly.
Winner split noteMUST when relevantIf output winner and business-understanding winner differ, state that split near the top instead of hiding it in history.
Proof-origin noteMUST when relevantIf the run that first closed the active truth is different from the retained output winner, keep that split visible near the top.
Deepest defended flow noteMUSTName the deepest business chain actually defended by evidence in this round, not just the active frontier label.
Exploration-versus-proof noteMUST when relevantSay explicitly whether broad exploration exceeded strong proof, so a reader can distinguish explored surface from defended business depth.
Short provenance noteMUST when relevantIf execution depended on bootstrap, borrowed generated support, or rebuilt execution scaffolding, keep the first-screen wording short and neutral, for example execution-layer rebuilt from explicit bootstrap. Do not dump long benchmark-memory prose into the hero summary.
2.2 Graphs, mapping, and coverage business model plus proof surface
SectionStatusWhat it must accomplish
Business Domain GraphMUSTModel the application as a business system, not as a test inventory.
Actor State GraphMUSTExpose state transitions, protected states, and blocked or guarded edges.
Business Flow GraphMUSTMap major journeys, entry points, goals, and completion logic in business terms.
Coverage VocabularyMUSTDefine covered, partial, deferred, blocked, regressed, variant, and excluded states in one explicit place so later tables and graphs do not silently drift between meanings.
Flow-To-Test-To-Support-To-Proof MappingMUSTShow how business flows map to generated support, generated test classes, and observable proof.
Validation MatrixMUSTMake engine/report proof depth explicit instead of burying it in review notes or run folders.
Accessible Hyperlink Exploration LedgerMUSTKeep the visible navigation frontier explicit, searchable, honest, and useful for next-frontier selection.
Route-Proof Versus Flow-Proof DeltaMUSTShow clearly when the round moved from route reachability into deeper business continuation such as autofill, payment, persistence, or success signals.
Truth-Ladder SplitMUST when relevantKeep route truth, protected-shell truth, flow truth, mechanism truth, and blocked truth visibly separate whenever guarded transitions or login-route volatility would otherwise flatten the story.
Business Flow InventoryMUSTTrack covered, partial, deferred, blocked, regressed, variant, and excluded journeys cleanly.
Coverage OverlayMUSTKeep covered, partial, blocked, regressed, and deferred states explicit and honest.
Deepest Defended Flow / Max Retained ChainMUSTShow the longest business continuation that remained defended by evidence, and name which kept run anchors that claim.
Exploration Breadth Versus Proven DepthMUSTSeparate what the round explored widely from what it actually proved deeply, especially when iteration count was high.
Deferred Or Tentative AreasMUSTExplain separately what remains deferred, tentative, blocked, or regressed and why those states were not promoted.
Disconfirmed Frontier NotesMUST when relevantExplain when a tempting frontier was actively disproved by automation, what evidence broke it, and why the round pivoted elsewhere.
2.3 Evidence, comparison, and memory why this run wins and what remains open
SectionStatusWhat it must accomplish
Cross-Run Lineage And Significant Run PackagesMUSTShow which kept runs matter, what they taught, and why the lineage remains explorable.
Validation Artifact ReferencesMUSTLink the exact kept run package, exact result XML or surefire proof, and any verified report artifact that supports the verdict.
Ledger readabilityMUSTKeep validation ledgers readable even when commands, paths, or artifact references are long: wrap safely, stack details, or collapse per-run detail instead of letting one wide field dominate the row.
Generated ATF Test-Result Quality ScoreboardMUSTTreat ATF trust as a first-class concern inside the analysis artifact itself.
Reporting Overall QualityMUSTExpose one explicit overall quality judgment for the reporting family, backed by visible reasoning.
Cross-Run ComparisonMUSTPrevent latest-run bias and explain promotion by evidence.
Checkpoint Self-EvaluationMUSTShow which checkpoint scores or benchmark rereads actually changed the next action, so the loop does not look like ten decorative runs with no steering logic.
Comparative Family SplitMUST when relevantSay explicitly which family changed business direction, which family improved only reporting/orchestration, and which family produced the retained output winner.
Proof-Origin / Retained-Close SplitMUST when relevantSay explicitly which run first closed or reconfirmed the active truth and which broader retained-close run became the strongest audit anchor.
Benchmark ComparisonMUSTName the relevant benchmark explicitly and explain where the current artifact is stronger, weaker, or merely different.
Local Exemplar ComparisonMUST when relevantIf a local exemplar library exists under docs/goldenCopies/html-exemplars-to-follow, state which exemplar reports were compared and whether the current report actually beat them.
Benchmark GapMUSTName the closest benchmark, where the current package is above it, and where it still remains below the stronger reference.
Frontier GapMUSTState the most valuable unfinished frontier in business language and why it matters more than cosmetic reruns.
HTML History IndexMUSTMake every semantic milestone explorable, descriptively named, and worth revisiting.
Iteration History And Score EvolutionMUSTShow how business understanding and trust changed over time.
Iteration Budget And Early-Stop LegitimacyMUSTState the planned iteration budget, the consumed depth, and whether any early stop was legitimate.
Effective Meaningful IterationsMUSTSay how many iterations materially changed business proof, evidence diversity, or benchmark/ frontier understanding. Decorative reruns, command-shape misses, and packaging-only touches must not be counted as meaningful depth.
Self-Perfection Audit TrailMUSTRecord what reusable lessons were pushed upstream versus what remained contextual to the current run.
Deferred Frontier / Recommended Next IterationMUSTKeep future work concrete, bounded, and honest; include the target gap when the target is not reached.
Render Verification StatusMUSTConfirm that the final HTML renders cleanly and the graphs remain visible and legible, then update the status rows to reflect actual verified, partial, or blocked state instead of leaving placeholder text.
Reporting failure: omitting a MUST section, shrinking it into one vague paragraph, or hiding it only in milestones means the canonical page is incomplete.
3) Presentation and density contract how the page should feel and scan
Core presentation rule: the canonical page should feel like a sober analysis dossier, not a poster or dashboard mockup.
  • The final page MUST use a consistent dark theme with restrained accents and stable contrast.
  • Typography MUST be compact and refined, but should read like a calm report rather than a dashboard: body text around 12.8-14px, tables around 11.5-12.8px, badges compact, and headline sizes restrained.
  • Typography MUST use one calm primary sans stack consistently across body, cards, tables, and graph labels. Do not mix legacy alternates such as Candara or Trebuchet MS into the canonical page when the standard stack is available.
  • The canonical page SHOULD prefer approximately: h1 in the 21-24px range, h2 in the 14.5-16.5px range, h3 in the 12-13px range, and a visually quieter lede than the main title.
  • Body text SHOULD usually stay near 13.2-14px with line-height around 1.58-1.64. Body sizes around 16-17px and oversized hero typography should be treated as legacy drift unless the page proves an exceptional readability need.
  • Score numerals and score cards MUST be readable but not aggressive, and should usually stay in the low-to-mid-20px band rather than dominating the first screen.
  • Score cards, summary pills, and hero blocks SHOULD share the same baseline font rhythm. If cards, body text, headings, or graph labels visibly feel like different design systems, normalize the CSS before treating the report as polished.
  • Above-the-fold space MUST be efficient: hero text, score chips, and trust framing should appear early without oversized empty space.
  • Vertical rhythm SHOULD be slightly more generous than a dense dashboard: paragraph spacing, table cell padding, and section separation should help scanning without creating poster whitespace.
  • Analytical density MUST come from distinct information layers: summary, graphs, mapping, comparison, history, hyperlink ledger, and deferred areas.
  • The page MUST NOT rely on giant cards, giant typography, oversized whitespace, or oversized SVGs to simulate quality.
  • Tables MUST be compact enough to scan but not so compressed that proof becomes unreadable.
  • Long command strings, file paths, selector chains, class lists, and dense proof details MUST NOT force horizontal overflow or visually broken rows in the canonical page.
  • When a ledger cell would become too wide, the report SHOULD prefer wrapped multiline code, stacked proof blocks, or a short inline summary plus a collapsible detail block instead of one giant unbroken line.
  • Monospace proof cells such as commands, paths, selector chains, and artifact references SHOULD render with wrapping-safe CSS such as white-space:pre-wrap, overflow-wrap:anywhere, and word-break:break-word when needed; a stable multiline cell is preferable to horizontal clipping.
  • Tables that still become too wide after sane wrapping SHOULD be split into coordinated narrower tables or converted into a compact ledger plus expandable per-run details.
  • Dense or secondary sections SHOULD be rendered as collapsible blocks that are closed by default unless they carry first-view truths needed to judge the run.
  • The closed state MUST be visually economical: one concise summary row, minimal padding, and no large near-empty shells.
  • The default open set SHOULD stay small and calm: the hero, core summaries, and the three primary graphs can remain visible, while proof-heavy ledgers and long histories should usually stay closed.
  • Hero styling SHOULD stay disciplined: moderate padding, restrained border radius, and no oversized title/lede pairing that makes the page feel like a separate visual system from the sections below it.
  • Collapsed sections SHOULD use short ASCII-safe summary hints when possible so the default closed view remains stable across renderers and encoding paths.
  • Proof Mapping, Validation Matrix, Iteration History And Score Evolution, Deferred Or Tentative Areas, Benchmark Comparison, Self-Perfection Audit Trail, Validation Artifact References, and long hyperlink ledgers SHOULD default to collapsed unless the page becomes less readable that way.
  • The report SHOULD keep one explicit primary retained validation run visible near the top so a reviewer can tell immediately which kept package anchors the verdict.
  • The report SHOULD keep pills discrete and semantic: a few useful truths are good, decorative score confetti is not.
  • The report SHOULD keep first-screen phrasing calm and slightly larger than earlier local drafts, but still sober and dossier-like rather than dashboard-aggressive.
  • Graph text SHOULD visually harmonize with body typography: avoid a graph-label style that looks heavier, larger, or more playful than the surrounding report.
  • The report SHOULD keep bootstrap or borrowed-support provenance near the top as one short pill or one short sentence only. Detailed bootstrap caveats belong lower in audit sections, not in the hero prose.
  • The top-of-page trust stance SHOULD stay shorter and more decision-dense than any provenance caveat. Readers should learn the artifact class, primary retained run, and main benchmark/frontier gap before they read deeper audit notes.
  • The first screen SHOULD not spend more vertical space on bootstrap or benchmark caveats than on the actual current baseline decision.
4) Diagram and render contract compact graphs, real render trust
Preferred baseline: static inline SVG graphs, because they remain stable across local file open, simple static hosting, and artifact promotion.
  • Diagrams MUST render without external dependencies.
  • Every canonical HTML MUST include a readable business domain graph, actor-state graph, and flow graph as distinct inline SVG blocks.
  • Graphs MUST be compact and fine-grained, not poster-style: thin strokes, concise labels, restrained node sizes, and minimal dead space.
  • SVG text inside graphs SHOULD use smaller sizes than surrounding prose, typically around 9.5-12px depending on hierarchy.
  • Primary node labels SHOULD be concise enough to avoid oversized boxes, and viewBoxes SHOULD stay tight.
  • Padding around SVG containers SHOULD be minimal and purposeful rather than poster-like.
  • Graphs MUST teach something different from the tables around them; they are not decorative restatements.
  • When the round promotes continuity across relogin, session clear, or another actor-state reset, the actor-state and business-flow graphs SHOULD show that boundary explicitly instead of implying only same-session depth.
  • Render verification is mandatory before promotion.
Graph failure: oversized diagrams, thick strokes, sparse labels, or giant node boxes that waste space and reduce analytical density.
5) Proof mapping and coverage vocabulary what is really proven and how to say it
5.1 Core proof semantics status words and mapping honesty
TopicExpectation
Coverage vocabularyThe canonical page MUST use explicit statuses such as Covered, Partial, Deferred, Blocked, and Regressed when those states are the truthful fit.
Flow mapping columnsThe main proof-mapping view SHOULD make business flow, status, generated support, test class, and observable proof visible in one place.
Proof-mapping width controlIf one wide table becomes hard to scan, the report SHOULD compact wording, split the mapping into coordinated tables, or use stacked proof blocks.
Long-line containmentLong commands, paths, class lists, selector chains, and artifact references MUST wrap cleanly inside their cell or move into stacked/collapsible detail blocks; the page should never depend on horizontal overflow to preserve fidelity.
Proof wordingObservable proof MUST name what was actually seen: UI text, route behavior, stable XML result, screenshot-backed report verification, or other concrete signal.
Generated-support visibilityGenerated support MUST be visible in the mapping, not hidden behind test names only.
Partial honestyIf only the entry contract or UI shell is proven, the flow MUST stay Partial or Deferred rather than being upgraded to Covered.
Visible-surface honestyIf a route or account surface is visible after a meaningful upstream success but no concrete record materializes there, the report SHOULD classify it as Partial or Deferred rather than upgrading it because the page exists.
Sub-contract split honestyIf a broader same-slice matrix proves only the core of a flow across engines or reports, the canonical page SHOULD split the claim into a covered core sub-contract and a partial or deferred richer sub-contract.
Regression honestyIf a flow failed during the run and was later restored, the final page SHOULD preserve that regression history instead of erasing it.
Coverage model rule: the page should help a reviewer answer "what is really proven?" without opening test classes first.
Width rule: proof mapping should stay scanable on a normal laptop viewport.
Good-versus-bad ledger rule: a good long command row wraps into a few calm lines inside the cell, or uses a short summary plus expandable details. A bad row stays as one unbroken line that stretches the table, clips content, or makes nearby columns unreadable.
5.2 Frontier depth and hyperlink interpretation beyond route names and shell proof
Hyperlink frontier rule: the ledger should not stop at route names. It should separate guest versus authenticated shells, note support and trust pages, distinguish internal versus external destinations, mark redirects, hidden descendants, duplicates, and visible product-card descendants, and explain which links changed trust, product map, side journeys, or frontier priority.
Depth-preference rule: once the report shows a shell or hyperlink frontier as stable, the next promoted section should prefer deeper observable business continuation when reachable, such as autofill, payment-method selection, success confirmation, persistence, or post-action continuity, instead of celebrating another shell-only hop.
Same-frontier rerun rule: if a later rerun proves the same frontier with cleaner package shape but without deeper business continuation, the canonical page should classify the delta as stronger retained proof or packaging gain, not as a new frontier.
Proof-origin / retained-close rule: when a round claims a family-level comparison, the canonical page should say which run first closed or reconfirmed the active truth and which run became the broader retained close. If the round only repeated one narrow command, say that the comparison stayed thin.
Max-flow rule: the canonical page should state the deepest defended chain in plain business terms, for example registration -> login -> profile -> messages -> invoices -> relogin continuity, and should not force the reader to infer that chain from multiple tables.
Iteration-depth rule: when the execution budget is high, the page should say whether later iterations increased defended depth, only broadened exploration, or merely hardened the same retained proof.
5.3 Compatibility and persistence honesty cross-engine slices and session-reset truth
Compatibility evidence rule: cross-engine, cross-browser, or alternate-reporting slices are supporting evidence unless they exercise the same promoted business contract and preserve machine-readable proof. Separate product truth from runner/process warnings; a Selenium or CDP warning is compatibility context, not product failure and not full-suite parity by itself.
Business-critical compatibility rule: after shell/link compatibility is stable, prefer cross-engine slices for promoted business-critical flows such as checkout/payment, account state, support propagation, recovery, or persistence before spending more effort on static-route reruns.
Session-reset rule: when the business claim is persistence rather than logout correctness, a cleared-session plus re-login proof may legitimately anchor the canonical truth more strongly than a flaky sign-out branch. The page should say so explicitly instead of implying that sign-out was the validated contract.
6) Milestone and history contract history as active memory
Milestones must read like rich case notes. They are not numbered breadcrumbs and they are not trimmed copies of the canonical page.
FieldExpectation
Milestone themeOne real shift in business understanding, frontier truth, stability, trust, or prompt quality.
DeltaWhat changed since the previous kept state.
Observed pressureWhat weakness, ambiguity, or failure triggered the iteration.
Fix or decisionWhat changed in prompt, code, or interpretation.
Score snapshotBusiness HTML, ATF Package, Overall, and Target status at that milestone.
Compact milestone shapeNew understanding, tests added, support added, evidence, score delta, and next gap.
Next frontierWhat should be attempted next if the loop continues.
Narrative contract: each milestone should say clearly what was discovered, what was confirmed, what was disproved, what was retained, and what moved next.
Canonical history contract: the main page SHOULD combine one compact score-evolution table with a more narrative history view so the history is not only numeric.
Dynamic memory contract: the next round should reread the canonical page, the milestone family, and meaningful prior run packages before deciding what to improve. History is active memory, not passive archive.
Pattern-detection contract: history exists to reveal repeated failure shapes, repeated misleading claims, recurring engine/report gaps, recurring hyperlink frontier misses, and promising next frontiers.
Message-to-artifact integrity contract: the user-facing verdict, canonical page, milestone family, and active run archive should all describe the same concrete round. If they drift apart, the report should downgrade trust and call the round audit-inconsistent.
Transient-stability contract: if an engine or report path first failed and then passed on an immediate same-slice rerun, the canonical history should keep both states visible and explain the lesson.
Run-lineage contract: the canonical page should make it obvious which preserved run packages are worth reopening and what each one contributes to the audit trail.
Run-numbering contract: when a workspace-local run archive is reset, the next local lineage should restart cleanly at run_01 and remain contiguous. External lineage can stay external, but inherited numbering inside a fresh local archive is an audit defect.
Winner-split history rule: if one run is the business-understanding winner and another run is the output or retained-package winner, history should preserve both roles explicitly instead of flattening them into one generic "best run".
Exemplar-beat rule: a report may be a valid round winner without being exemplar-grade. If the page did not clearly beat the local exemplar library on trust, proof mapping, benchmark/frontier explainability, artifact references, and readable audit density, the report must not be copied into docs/goldenCopies/html-exemplars-to-follow.
Hard failure: milestone families made of near-identical "still green" pages, or a long run that preserves only one thin summary instead of an explorable milestone family.

7.2 Verification and evidence integrity render, hydration, screenshots, references
Early-stop truth rule: if the loop consumed fewer iterations than planned, the canonical HTML must justify that exception explicitly against open seed-adjacent flows, meaningful hyperlinks, and the benchmark gap.
Render-verification rule: completion requires actually opening the generated report and judging readability, graph proportion, table width, heading aggression, and scanability.
Hydrated-render rule: when a referenced report bundle is JS-driven, visual verification is valid only after the report is shown in a hydrated state.
Visual-evidence rule: the run package should preserve at least one screenshot or comparable visual artifact for the verified canonical page and for any benchmark-significant report comparison that influenced the verdict.
Verification-status rule: once screenshots or equivalent proof are captured, the canonical page should replace pending render placeholders with explicit verified status and point to the preserved evidence paths.
Validation-reference rule: the canonical page must include a dedicated Validation Artifact References section that links directly to the kept run package, the exact XML or surefire proof, the preserved Extent report when relevant, and the preserved Allure report only when that Allure artifact was visually verified in a hydrated state.
Ledger-readability rule: before calling the page reporting-complete, perform a quick pass for long command, path, and artifact rows: no horizontal overflow, no clipped command cells, and no single unwrapped field dominating the row.
Primary-run alignment rule: the final page must point to one explicit primary retained validation run and keep that choice consistent across the summary, lineage, and validation-reference sections.
Source-lock reporting rule: when the round compares families or borrowed standards, the canonical page should list exact files used, exact files excluded, and any explicit WIP exclusion that governed the round.
Bootstrap-provenance placement rule: if the round reused generated support, rebuilt an execution layer from a benchmark snapshot, or depended on a shared precursor bootstrap, the page should keep only a short first-screen provenance note and move the detailed explanation into Reset And Source Lock, Validation Artifact References, or a dedicated provenance block. Do not frame bootstrap provenance as if it were a business benchmark victory.
Benchmark-versus-bootstrap rule: distinguish benchmark memory reopened for comparative judgment from execution bootstrap provenance. The first changes how the round was judged; the second explains how the round became runnable. Do not collapse them into one generic benchmark-memory sentence.
Output-versus-business rule: when the retained output winner differs from the business-understanding winner, the page should say that explicitly in the comparative summary and cross-run comparison.
Proof-origin memory rule: when the run that first closed the truth is different from the retained output winner, the page should keep both visible instead of letting the later cleaner package erase the origin of the business proof.
Report-log preservation rule: if a log or preserved report explains a repair, engine difference, hydration issue, or final trust claim, the canonical page should link it from the validation references instead of leaving it trapped only in terminal memory or transient target output.
Blocked-visual rule: if local rendering is blocked by file policy, hosting limitations, or incomplete hydration, the page must downgrade render verification to Partial or Blocked and link the blocker note or screenshot instead of pretending success.
Checkpoint-self-eval rule: the canonical page should make visible how comparative self-evaluation against the direct predecessor and primary benchmark changed the round direction, not just the final score.
Final readability checklist: no horizontal overflow in proof-heavy tables; no visibly clipped command or artifact cells; long-path and long-command rows remain readable on a normal laptop viewport; if not, wrap, stack, split, or collapse before promotion.
7.3 Frontier-specific honesty rules hard cases that must stay explicit
Password-proof rule: when a round claims deeper password lifecycle coverage, the report should state whether it proved only a success confirmation, or the stronger contract of old-password invalidation plus new-password relogin, and score the frontier accordingly.
Profile-update rule: when a round explores profile-edit persistence, the report should state whether changed values actually survived save plus reload. If the page accepts edits but reverts to the prior values, classify the frontier as blocked or deferred instead of implying persistence.
Disconfirmed-proof rule: when automation shows that a visible UI surface throws a client error, issues no real request, or fails stronger state-transition proof, the report should classify that frontier as blocked or deferred rather than preserving the earlier manual optimism.
Displaced-frontier reporting rule: if the direct predecessor was already beaten on a shallower frontier, the canonical page should say so explicitly and explain which deeper benchmark gap replaced it as the round's main story.
Compare-proof rule: when comparison is promoted, the report should show selection-state proof, comparison-table proof, at least one divergent attribute proof, and any empty-state or clear-all recovery proof used to justify the claim.
Support-continuity rule: when a support or contact action continues into a protected message center, detail route, or reply thread, the canonical page should promote the persisted protected continuation as the real frontier rather than stopping at the public success confirmation.
Session-continuity rule: when the new retained truth depends on relogin, session clear, or another actor reset, the canonical page should make that boundary visible in the graphs, proof mapping, and benchmark comparison instead of burying it only in one sentence.
Borrowed-family rule: if external standards improved only provenance, source lock, or packaging clarity, the page should say that directly and avoid implying that borrowed prompts created the business frontier.
Bootstrap honesty rule: when a shared-engine precursor remains blocked and the round advances only through an explicit bootstrap or workaround, the report should say where the real blocker still lives instead of implying a fully native path.
Pill-value rule: the pill rail should carry a few immediately useful truths such as primary run, baseline count, explicit non-promotion, or benchmark gap. If a pill does not help a reviewer judge trust quickly, it should not be there.
Self-perfection rule: the canonical page should end with a clear trail of what stable lessons were upstreamed into shared prompts or standards, what stayed local, and why.
8) Promotion and upstream integration when this artifact is strong enough to reuse
  • The canonical HTML MUST explain why the current run won when multiple governed runs exist.
  • The canonical HTML MUST keep deferred areas explicit and honest.
  • The canonical HTML MUST be rich enough to become a reference exemplar without requiring separate chat memory.
  • The execution prompt should reference this reporting contract instead of cloning it inline.
  • Only reusable reporting rules should live here; run-specific discovery, validation, and frontier logic should stay upstream.
  • The execution and meta prompts should point to the same canonical sections, presentation expectations, milestone expectations, validation-reference expectations, and trust rules.
Promotion bar: a canonical HTML can become a reference exemplar only if a new reader can understand scope, trust level, benchmark relation, generated ATF confidence, evidence paths, and remaining frontier without opening any additional artifact first.
Golden-copy bar: copying into docs/goldenCopies/html-exemplars-to-follow is allowed only when the current report beats every relevant stored exemplar overall, not just when it wins the current round. A prettier page, a narrower local frontier win, or one stronger subsection is not enough by itself.
