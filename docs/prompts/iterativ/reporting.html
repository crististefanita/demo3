Standard de raportare scheletul canonic al paginii finale

Acest document defineste forma canonica pentru analysis/langgraph-business-understanding.html. Raportul final publica noutatea business, dovada locala, accounting-ul si adevarul despre miscarea urmatoare, fara sa reinterpreteze legea de launch sau artefactele decisive din core.
business-first proof mapping dark + collapsible

Overview

Detine
forma finala, ordinea de lectura si hyperlink-urile locale
Nu detine
accounting-ul de productie, valorile active si legalitatea
Publica
noutatea business, dovada, accounting-ul inghetat si miscarea urmatoare
Cere
graph lizibil, proof mapping, artifact index, run audit si close-out clar
Fiecare claim important trebuie sa aiba hyperlink local catre dovada lui.
Contract minim:
Raportul nu defineste valorile active.
Raportul nu defineste accounting-ul rece.
Raportul nu defineste legalitatea.
El publica separat valorile din runtime, accounting-ul din orchestration si verdictul din core.
Daca publicarea suna mai sigura decat freeze-ul, raportul este gresit.

Structura obligatorie

Sectiune Ce trebuie sa contina
1. Deschidere hero verdict, runtime strip, 4 score cards si accounting compact
2. Verdict executiv winner truth, strongest unresolved truth, same-round truth, fresh ROOT_RUN truth si mutatia istorica fata de pachetul precedent
3. Graful fluxului business macro-harta business reala a aplicatiei, apoi overlay-ul pachetului: baseline covered, new frontier, owned, partial, blocked, fresh-round-only
4. Dovada si acoperire proof mapping, artifact index si orice subsectiune utila ramasa hyperlink-uita
5. Banda de audit pe run o banda scurta per run: claim, delta, kept, diff, Extent, feedback
6. Adevarul de inchidere x / ROUND_TARGET_TEST_COUNT, runs consumed = n / META_ITERATION_COUNT, target met, same-round legality, fresh ROOT_RUN legality, open problems, under-spend explicit fata de valorile active publicate
7. Randare si audit pagina canonica, shared CSS, render proof, link audit, collapse discipline
1. Flow-uri si diagrameforma si dovada paginii finale
1.1 Opening shapeforma primului ecran
Legenda
  • Nimic standalone intre cards si sectiunea urmatoare.
  • Accounting trebuie integrat in score wall, nu helper slab separat.
  • Jump link-urile sunt bune doar daca nu umplu opening-ul cu zgomot.
1.2 Business Flow Graphharta aplicatiei + overlay-ul pachetului
Legenda
  • Owner-ul grafului este aplicatia, nu pachetul.
  • Primul subgraf obligatoriu este macro-harta business reala a aplicatiei, inclusiv lane-uri relevante neatinse in acest pachet.
  • Overlay-ul pachetului curent se deseneaza doar peste acea harta: ce a fost atins, owned, partial, blocked sau fresh-round-only.
  • Nodurile spun povestea business; detaliul de test coboara in legenda, proof mapping si run audit.
  • Daca graful poate fi citit ca jurnal de frontier ranking sau lista de familii testate, este invalid.
1.3 Claim -> proof -> artifactlantul minim al publicarii
Legenda
  • Fiecare claim publicat trebuie sa aiba owner si dovada navigabila.
  • Daca lipseste ownerul, diff-ul sau proof carrierul, claim-ul nu e rece.
  • Claim-ul publicat trebuie sa sune mai sus decat numele testului care il sustine.
1.4 Adevaruri care nu trebuie amestecatecountability, runs consumed, same-round, fresh-root
Legenda
  • Aceste patru verdicte sunt diferite chiar daca apar in acelasi close-out.
  • same-package blocked nu inseamna automat fresh ROOT_RUN legal = no.
  • Daca aceste adevaruri sunt amestecate, raportul suna mai sigur decat este.

4. Graful fluxului businesscentrul pedagogic al raportului

fresh-round-onlyce ramane valid dar apartine altui pachet
Regula de prezentare Concret
Owner al grafuluiaplicatia reala; pachetul curent este doar overlay peste harta ei
Forme permisediagrama, business lanes sau 2-4 subgrafuri mici
Gramatica preferataactor -> intentie -> pas vizibil -> owned / blocked / fresh-round-only
Text in nodurititlu scurt + maxim o linie scurta de context
Legendascurta, colapsata, sub fiecare subgraf dens; muta acolo detaliul de test, run ids, nume de metoda si repair trail
Stilrefoloseste clasele si culorile din shared-report-reference.css
Nu facenu dubla proza sectiunii in noduri, nu umple legenda cu eseu si nu lasa graful sa sune ca jurnal de executie al pachetului
Daca graful devine greu, sparge-l in subgrafuri mai mici. Claritatea business bate ambitia vizuala.
Daca cititorul retine run_03, repair, reentry, nume de produs sau nume de test mai repede decat lane-ul business al aplicatiei, graful trebuie racit.
Aplicatia este harta. Pachetul este doar overlay-ul care coloreaza ce a fost atins, owned, partial sau ramas deschis.
5. Dovada si acoperiremapping, diff, Extent, artifact index
5.1 Mapare ownership si dovadaclaim -> run -> proof
Coloana Ce trebuie sa existe
Business claimclaim rece despre un pas sau o identitate business, nu doar despre un test verde
Run ownerlink la runs/run_##/SUMMARY.md
Evolution truthlink la runs/run_##/review/thin-no-index.diff pentru claim-ul de run; root-source-thin-no-index.diff doar pentru sinteza de pachet
UI prooflink la ExtentReport_*.html pentru pachetele UI
Machine prooflink la XML sau command-result.md
Cold verdictowned / partial / blocked / fresh-round-only
Pe langa linkul la diff, fiecare claim important trebuie sa aiba o propozitie scurta despre mutatia istorica adusa: ce s-a deschis, ce s-a inchis sau ce contradictie s-a ingustat.
Daca claim-ul cere interpretare suplimentara, linkul la source-delta-carrier.md sta langa diff, ca suport. Nu ii ia locul.
Fara link la diff-ul mic al run-ului, claim-ul publicat nu are istorie rece completa chiar daca root diff-ul exista.

6. Banda de audit pe runo banda rece per run
Coloana Continut minim
Runrun_## + link la SUMMARY.md
Business claimce a incercat sa apere si de ce merita consumat acel run
Expected deltace fel de noutate sau adancime se astepta
Actual deltace s-a materializat rece si ce mutatie istorica a adus fata de starea anterioara
Kept?yes / no
Why countable / not countablemotiv rece, scurt
Difflink la diff-ul mic relevant al run-ului; nu doar la root diff
Extentlink pentru UI sau n/a explicit
Feedbacklink la feedback-ul constructiv salvat
Daca un raport are doar root diff si nu te duce clar la deltele mici per run, istoricul lui este auditabil partial, nu profund.
Banda pe run trebuie sa arate direct: claim -> run -> run diff -> Extent/XML -> verdict. Orice alta ordine face istoria mai lenta.
Daca un run a schimbat sursa, dar randul lui nu duce la runs/run_##/review/thin-no-index.diff, randul trebuie tratat public drept incomplet, nu doar slab.
Banda pe run trebuie sa permita doua lecturi rapide: ce a dovedit run-ul si ce a schimbat el in istoria pachetului.

Nu amesteca intr-o singura propozitie countability, breadth, runs consumed, yield si application exhaustion. Contoarele raman separate.
Raportul doar publica rezultatul de accounting si close-out deja decis in orchestration. Nu muta aici teoria despre cum se produce targetul; doar arata clar ce a iesit si care a fost verdictul final.
