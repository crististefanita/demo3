Prompt Evolution Orchestration Standard shared standard

Reusable orchestration contract for prompt-evolution workflows. This file standardizes only the durable supervision shape: preserve-reset-rerun discipline, run-package integrity, winner selection by evidence, promotion trust, and upstream ownership boundaries. Repository-local thresholds, path roots, loop depth, execution commands, and family-specific stop policy must stay upstream.

Preserve-reset-rerun discipline, run-package contract, failure patterns, cross-run winner logic, and promotion expectations. Loop depth, quality targets, path roots, reset scope, comparison scope, family-specific stop rules, and any local exemplar-library path.
Goal: every governed loop should preserve evidence, improve the governed prompt family when needed, reset only superseded generated output, rerun from a cleaner state, and promote the strongest run by evidence rather than chronology.
Use this standard when: you want multiple prompt families to share one honest orchestration model instead of reinventing preserve/reset/rerun and winner selection every time.
1) Purpose and adoption contract what this standard is for
1.1 Core purpose why the file exists
  • Preserve every governed run as a reviewable evidence package.
  • Require prompt evolution when weaknesses are traced to wording, structure, or ownership.
  • Reset only superseded generated output, not the kept evidence family.
  • Force cross-run winner selection by evidence, not chronology.
Concrete supervising prompts should stay directly executable, but they should reference this file instead of cloning the generic orchestration contract inline.
1.2 Layer ownership who owns what
  • The execution layer should own discovery, generation, and run-local artifact production.
  • The supervising layer should own scoring, prompt evolution, rerun discipline, and final promotion.
  • Future prompt families should improve this shared contract directly when the reusable orchestration model gets better.

2.3 Retention and lineage discipline how archives stay honest
  • A losing run still MUST remain reviewable enough to explain why it lost.
  • The loop MUST NOT rely on terminal output alone as the preserved memory of a run.
  • Every preserved run MUST be intelligible in under one minute from its SUMMARY.md; folder presence without a clear audit explanation is not sufficient evidence.
  • Run retention MUST preserve every real run with a distinct defect, repair, rerun, strategy change, benchmark comparison, or reusable lesson. Do not compress meaningful lineage to only the last two or three runs; do delete or complete decorative run folders that add no new evidence.
  • The final retained validation folder MUST be clean: keep the final verdict report and exact supporting logs/evidence, and move unrelated earlier reports to their own run packages instead of mixing them into the final folder.
  • Fresh family-comparison or quality-loop validation rounds SHOULD live under one explicitly named active-scope subfolder beneath evolution-runs rather than dumping new run_01..run_## directly into a root that may later mix lineages.
  • If the loop continues over retained run packages instead of rebuilding from an empty archive, the SUMMARY and canonical report MUST call it a continuation round, list inherited runs, list new runs, and avoid stale wording from previous run counts or budgets.
  • The loop MUST NOT inflate run count by alternating engines or report adapters over the same unchanged hypothesis. A counted run needs new proof, exposed defect, confirmed repair, changed trust, or a documented blocked/low-value frontier.
  • The loop MUST NOT present four identical reruns of the same narrow command as a sufficient full-family comparison unless the review proves that no broader symmetric retained-close evidence class was feasible.
  • The latest run is not automatically the best run.
3) Improve, reset, rerun, and failure rules how the loop stays honest
3.1 Core rerun sequence the step order
  1. Review the active run package.
  2. Identify prompt-caused weaknesses in analysis quality, reporting quality, generated output quality, or history usefulness.
  3. Classify each lesson as reusable prompt knowledge or contextual run evidence before changing shared standards.
  4. Improve the relevant governed prompt or shared standard reference.
  5. Preserve the full run package.
  6. Reset only superseded generated output.
  7. Rerun from a cleaner state.
  8. Score again and compare against the best-so-far run.
3.2 Reset and learning rules how resets and lessons stay trustworthy
Reset rule: preserve analysis memory, review evidence, manifests, and prompt history first; revert only generated output and generated support that were superseded by the prior run.
Reset-proof rule: when a round claims a fresh reset, preserve a small reset-evidence note in the active scope that records what was reverted or removed, what remained intentionally, and a concise workspace status check before the first kept run.
Self-perfection audit rule: every standard or prompt improvement should be traceable to a run package, failure, validation gap, render inspection, hyperlink-frontier discovery, or benchmark comparison. If no such evidence exists, keep the observation local instead of changing the standard.
Dynamic-history rule: significant prior runs must be reread for repeated failures, repaired weaknesses, score drift, benchmark gaps, unresolved frontiers, and reusable engine/report lessons before the next rerun plan is declared. History is active memory, not passive archive.
Prompt snapshot learning rule: when benchmark packages contain the prompt snapshots that produced strong results, inspect those snapshots as first-class evidence before changing shared standards. Learn the reusable execution or supervision rule that produced the trust improvement; do not copy local product wording or promote style-only differences.
3.3 Special comparison cases how to classify tricky reruns
Targeted-rerun classification rule: when a full retained-candidate suite fails and a targeted rerun later passes, classify the targeted pass as diagnostic, repair, or flake-isolation evidence. It may guide the next action, but it MUST NOT replace the failed full-suite verdict or become the promoted baseline until a clean full rerun confirms the whole candidate package.
Proof-origin / retained-close rule: when a round claims a full-family comparison, each family SHOULD contribute one proof-origin checkpoint and one broader retained-close checkpoint. If the run only reconfirms the same narrow command, call it repeatability evidence, not mature family comparison.
Sub-contract refinement rule: when a broader same-slice matrix shows that one engine or report path proves only the core of a business contract while another proves richer fields, refine the promoted claim into covered core proof plus partial or deferred sub-claims instead of forcing false equivalence.
Transient evidence rule: a first-fail then pass sequence is not noise by default. Preserve it when it reveals timing, hydration, selector, or environment sensitivity; only downgrade it to contextual noise when the run package proves no trust decision changed.
Winner rule: the latest run is not automatically the winner. Promotion must explain why the selected run displaced the previous best and why the losing runs were not promoted.
3.4 Failure modes to reject invalid or misleading loop behavior
Failure mode to reject Why it is invalid
Prompt weakness was diagnosed, but no relevant prompt evolution happened before rerun.The rerun is not a meaningful test of improvement.
The run passed validation, but the canonical analysis artifact is too thin to justify trust.Passing validation alone does not make the package trustworthy.
The run improved breadth but weakened determinism or honesty.Coverage growth does not excuse weaker trust.
The loop preserved artifacts, but not enough evidence to explain why a previous run lost.Cross-run promotion becomes decorative instead of evidence-based.
The loop claimed a clean rerun even though reset safety allowed only a partial rollback.The claimed rerun state is misleading.
The loop called a family comparison complete even though one or both families lacked a broader retained-close checkpoint.The comparison stayed too thin to justify a strong family-level verdict.
4) Quality gates, promotion, and reference rules what must be judged before promotion
4.1 Quality gates what must be scored
Gate Expectation
Business HTML qualityThe run-local canonical analysis artifact and semantic history must be truthful, rich, rendered correctly, and useful as reusable business evidence.
Reporting overall qualityThe reporting family must be strong enough to support onboarding, trust, and winner selection.
Generated output qualityThe produced generated output must be stable, meaningful, maintainable, mapped to the analysis artifact, and validated by the expected evidence model.
Prompt baseline qualityThe governed prompt baseline must have improved if the prior run exposed prompt weakness.
4.2 Promotion rules how a winner gets promoted
  • The loop must score all gates; a run is not best merely because one gate is strong.
  • The winner MUST be justified by analysis quality, reporting quality, generated output quality, history usefulness, and trust honesty.
  • The winner MUST be promoted explicitly into canonical locations and, when useful, into reference exemplars.
  • Promoted reference exemplars MUST be copied only from the selected winner.
  • Promoted reference exemplars under docs/goldenCopies/html-exemplars-to-follow MUST be copied only after the supervising quality-loop proves that the final report beats every relevant exemplar already stored there.
  • A local round winner that does not beat the current exemplar library MUST stay a preserved winner without exemplar copying.
  • A promoted baseline MUST remain traceable back to the winning run package that justified it.
  • The promotion note MUST classify the artifact honestly as benchmark-grade winner, stronger local candidate, validated learning candidate, or non-promotable retained package. If a stronger benchmark still exists, the note MUST say so explicitly.
  • The promotion note MUST say whether the winning run is also the proof-origin run, or only the strongest retained-close package.
  • The promotion note MUST name the displaced predecessor when one exists, and explain whether the new winner changed business depth, only absorbed prior proof more broadly, or only improved orchestration/reporting discipline.
  • Prompt-only improvement MUST NOT be promoted as generated-output success until a later execution proves better output evidence.
  • Visual/reporting-only improvement MUST NOT create a promoted generated-output candidate unless paired with executable evidence that improves or stabilizes a meaningful frontier.
Promotion bar: a run deserves promotion only if a later reviewer can understand why it won, what it proves, what it still defers, and which evidence package justified it without reconstructing the decision from chat or terminal logs.
Golden-copy bar: exemplar copying is stricter than normal promotion. The selected winner must beat the local exemplar library overall, not merely win the current round or look visually cleaner.
5) Upstream ownership and consistency gate keep thresholds out of the standard
5.1 Ownership boundaries what stays shared vs local
Ownership rule Meaning
The standard defines loop behavior, run-package shape, failure patterns, and promotion rules.Use this file to declare what the workflow must preserve, compare, and justify.
The calling supervisor defines numeric targets, loop depth, path roots, and reset scope.Use upstream documents to decide how strict the governed family should be.
The kept review artifacts may display thresholds or targets.Those values should come from the governing prompt family, not from this shared standard.
Parameter rule: keep the shared standard intentionally small. Reusable parameter groups are enough; numeric thresholds, formulas, exact naming patterns, and family-specific stop severity stay upstream.
