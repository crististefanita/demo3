Prompt Evolution Orchestration Standard shared standard

Reusable generic execution contract for one governed run_## at a time. This file governs saved run packages: what must be preserved, how a kept run stays reopenable, how closure stays honest, and how saved artifacts stay comparable across reruns. Family choice, winner promotion, and broader benchmark pressure stay upstream.

9.2 Run-quality threshold and formula judge one run by verdict-moving delta, not by volume 
quality-control schema
  -> primary run gate
     RUN_EXPECTED_DELTA_QUALITY

  -> computed run result
     RUN_DELTA_SCORE_ACTUAL

  -> secondary round accounting
     ROUND_TARGET_TEST_COUNT

  -> active governed-run window
     META_ITERATION_COUNT
RUN_DELTA_SCORE_ACTUAL
  = 10 * (
      0.30 * T
    + 0.25 * B
    + 0.20 * P
    + 0.15 * H
    + 0.10 * C
    )

where
  T = truth delta
  B = branch / depth delta
  P = proof strength delta
  H = benchmark / promotion honesty delta
  C = closure trust delta

clear the run-quality bar only when
  RUN_DELTA_SCORE_ACTUAL >= RUN_EXPECTED_DELTA_QUALITY
  and T >= 0.67
  and B >= 0.50
  and P >= 0.75
Scoring item Role or weight Audit baseline or stance
RUN_EXPECTED_DELTA_QUALITYPrimary run gate.The run must clear this before it reads as verdict-moving.
RUN_DELTA_SCORE_ACTUALComputed support behind the gate.Do not invent a fuzzier overall-quality constant above it.
ROUND_TARGET_TEST_COUNTSecondary accounting target.Tracks breadth converted into tests, not winner quality by itself.
META_ITERATION_COUNTGoverned-run hard cap and loop-pressure guard.Constrains the active package window, not just the score.
T truth delta0.30Legacy tests plus the strongest prior kept run.
B branch / depth delta0.25The strongest prior kept run on the same branch family.
P proof strength delta0.20The prior proof chain for the same claim.
H benchmark / promotion honesty delta0.15The prior promotion-risk state and prior baseline decision.
C closure trust delta0.10The prior closure state of the same round family.
9.3 Test-materialization quality judge the added tests themselves, not the whole run 
TEST_MATERIALIZATION_QUALITY_SCORE
  = S + A + O + M

where
  S = semantic flow relevance      (0.0 - 3.0)
  A = assertion novelty / value    (0.0 - 3.0)
  O = observability / proof fit    (0.0 - 2.0)
  M = maintainability discipline   (0.0 - 2.0)
Component Audit question
SDo the added tests defend a flow, branch, recovery, or guarded continuation that matters to the business model?
ADid they introduce observable assertions that materially extend what the round can claim?
OAre the assertions visible, strongly anchored into proof artifacts, and tied to a cold-distinct target identity?
MAre the tests organized and named in a way that preserves long-term meaning instead of brittle or duplicated coverage?
Target-identity cap: when reopening or rereading depends on preview text, row snippets, repeated cards, or other possibly colliding surface fragments, O cannot exceed 1.0 / 2.0 until the run preserves a cold-distinct target anchor.

13) Failure, rerun, and comparison honesty how reruns stay harsh, comparable, and non-inflated
13.2 Failure classification and honesty classify the failure before deciding rerun, package repair, or closure
Failure class Recognize it when Required move
Execution failureThe run never reached stable execution for the intended scope.Publish exact blocked state and decide whether another governed run is still justified.
Proof failureThe run may have moved truth, but the saved validation bundle cannot reopen it cold.Do not promote; repair proof or rerun honestly.
Closure failureThe run looks final, but package or root carriers are still missing or stale.Finish package closure before calling it retained.
Canonical closure failureThe canonical package page changed but render proof or root/package closure stayed stale.Rebuild canonical closure and render truth before close-out.
Memory failureHistory or run memory lost causality, corrected assumptions, or open pressure.Restore the missing memory before compressing further.
 
boundary
  proof failure   -> decisive bundle missing
  closure failure -> decisive bundle exists, root/package closure missing or stale

13.4 Anti-inflation counting what deserves preserved weight and what must stay only lineage
Counting lens Keep it this way
Governed family accountingCount governed runs kept, intermediate attempts, meaningful iterations, and verdict-changing iterations explicitly.
Run-count anti-inflationA counted run needs new proof, exposed defect, confirmed repair, changed trust, or documented blocked frontier truth.
Same-hypothesis replay guardRepeat a narrow hypothesis only while it still teaches causal truth, trust movement, or repair value.
Latest-run bias guardThe latest run may win only if preserved evidence shows real displacement of the previous best.
Same proof repackagedDoes not count as a new governed run or truth move.
Packaging-only wording changeCounts only as packaging repair, never as executable progress.
Same command rerun, no new truthMay support trust, but does not count as a new governed gain.
New validation branch, distinct evidenceMay count when it creates a new reopenable claim or materially changes trust.
New kept run after changed frontierCounts when the frontier, evidence, or honest cap really changed.
