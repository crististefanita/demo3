Standard de orchestrare a evolutiei de prompt legea artefactelor si a invatarii

Standardul pentru artefacte salvate: aici se produc, se ingheata si se explica testele, diff-urile, dovada UI si invatarea rece a pachetului. Core decide legalitatea. Raportarea doar publica.

Overview

Detine
productia testelor, freeze-ul, diff-first truth, dovada si accounting-ul pachetului
Nu detine
legalitatea launch-ului, numarul de run-uri permis si layout-ul final
Judeca
gruparea, countability si ce ramane partial sau under-spent
Regula
orice adevar care poate schimba urmatorul run trebuie salvat auditabil
Contract minim:
Orchestration citeste valorile active din runtime si le trateaza ca lege a pachetului curent.
ROUND_TARGET_TEST_COUNT = minimul de teste bune auditabile pe care fiecare run trebuie sa il produca si sa il inghete.
META_ITERATION_COUNT = cele N run-uri pe care pachetul trebuie sa le consume si sa le inghete slot cu slot.
Teste bune = teste countable: auditabile, distincte business, sustinute de proof; nu doar verzi.
Cele doua contoare nu se compenseaza si nu se amesteca.
Sub oricare valoare activa: defect procedural sau blocker sever documentat.

Citire scurta: runtime-ul decide valorile. Orchestration produce si ingheata. run diff = mutatia mica a run-ului. root diff = mutatia compilata a pachetului. source-delta-carrier.md = suport doar daca diff-ul singur nu ajunge.
1. Boundary-ul orchestrationlegea artefactelor, nu legea de launch
DetineNu detine
inventarul de artefacte, freeze bundle, adevarul source diff-first, proof carriers, MD-urile de invatare, memoria de pachet, linkabilitatea ready-for-reportlegalitatea frontierelor, numarul de run-uri, legea continuarii in acelasi pachet, layout-ul final al raportului
Agentul poate alege liber implementarea, probele si notele auxiliare. Orchestration cere doar ca dovezile si lectiile decisive sa fie salvate rece. Daca run-ul sau pachetul raman sub valorile active, explicatia sta aici, in quality-accounting-verdict.md si package-close-out.md. Core doar verifica legalitatea, iar raportarea doar publica.
2. Flow-uri si diagrame banda artefactelor
2.1 Split runtime -> core -> orchestration -> raportare cine fixeaza, cine decide, cine publica
Legenda
  • Runtime-ul fixeaza configuratia comuna; nu decide singur nici legalitatea, nici publicarea.
  • Core decide daca urmatorul slot este legal.
  • Orchestration produce artefactele, explica progresul si freeze-ul pachetului.
  • Raportarea doar publica adevarul deja inghetat.
2.2 Ciclul unui singur run ce trebuie sa faca fiecare run
Pentru legalitatea launch-ului si a continuarii, vezi iterative-core-standard.html. Pentru forma finala publicata, vezi reporting.html.
Ce trebuie sa ramana in minte
  • Redeschide = recitesti ce poate schimba verdictul, nu faci warm restart.
  • Pretrain = fixezi frontiera si ce inveti din benchmark-uri si memorie locala.
  • Executa = exact un slot.
  • Un slot produce si explica targetul activ de teste bune auditabile; daca nu il atinge, verdictul este defect procedural sau blocker sever.
  • Daca ai ajuns intr-o pagina, stare sau functionalitate bogata, grupezi in acelasi run un set coerent de teste inrudite.
  • Dupa atingerea lui tests produced, run-urile ramase se consuma doar pe cea mai grea continuare legala sau se inchid prin blocker sever documentat.
  • Ingheata = fara diff, Extent/XML si MD-uri decisive, run-ul nu exista rece.
  • source-delta-carrier.md este obligatoriu doar cand diff-ul singur nu este suficient de auto-explicativ pentru claim-ul publicat.
  • Invata + preda = urmatorul agent trebuie sa poata redeschide pachetul fara chat.
2.3 Regula de grupare in acelasi runcand o zona merita mai multe teste
Daca ai ajuns aiciJudeciForma buna
o pagina / stare / functionalitate cu mai multe comportamente business distinctedaca aceeasi ancora poate produce cel putin targetul activ fara sweep mecanicgrupezi testele inrudite in acelasi run pana atingi minimul cerut pe run
o ancora care nu poate sustine targetul activdaca lipsa vine din blocker real sau din alegerea gresita a frontiereinu accepti run-ul ca valid slab; publici defect sau blocker sever si rerank
o zona care cere alta identitate business sau alta dovadadaca separarea ajuta claritatea, izolarea sau rerank-uldesparti in alt run doar cand tot poti atinge targetul activ in noul run
Legenda
  • Exemplu bun: aceeasi zona favorites poate produce in acelasi run empty-state, persistence si collection behavior, daca toate vin din aceeasi ancora functionala.
  • Exemplu prost: cinci variante apropiate ale aceluiasi click sau ale aceluiasi blocker doar ca sa umfli numarul.
  • Gruparea buna se face pe functionalitate si adevar business, nu pe comoditatea implementarii.
  • La finalul fiecarui run, orchestration trebuie sa spuna explicit daca targetul activ pe run a fost atins; daca nu, verdictul este defect sau blocker sever.
Detalii de executieinainte / in timpul / dupa run

Inainte de executie:

  • reciteste benchmark-urile deja selectate in pretraining;
  • reciteste artefactele relevante;
  • reciteste feedback-ul si lectiile salvate;
  • reciteste run-urile relevante;
  • produce pre-training;
  • recalibreaza frontiera business.

In timpul executiei:

  • urmareste business nou justificabil;
  • maximizeaza valoarea iteratiei curente si atinge minimum targetul activ de teste auditabile bune pe run;
  • daca o ancora functionala sustine mai multe teste inrudite bune, le grupeaza in acelasi run si le explica drept familie coerenta;
  • daca ancora nu sustine targetul activ pe run, nu inchide cald; publica defect sau blocker sever si rerank;
  • citeste constanta si ca obligatie minima pe run, nu doar ca target de pachet;
  • nu foloseste libertatea de grupare ca scuza pentru sub-target; run-ul trebuie sa atinga targetul activ sau sa publice defect / blocker sever;
  • judeca mereu explicit doua axe diferite: cate teste bune a scos run-ul si daca acel run merita slotul consumat;
  • produce dovezi verificabile;
  • materializeaza intelegerea business in teste si artefacte.

Dupa executie:

  • salveaza diff-urile;
  • salveaza XML-urile si rezultatele;
  • salveaza Extent pentru UI;
  • salveaza feedback-ul constructiv;
  • salveaza lectiile invatate;
  • scrie rece in accounting daca pachetul este inca sub ROUND_TARGET_TEST_COUNT sau sub META_ITERATION_COUNT;
  • lasa suficiente artefacte pentru audit si reluare la rece.
2.4 Lantul artefactelorcum devine un run redeschis rece
2.4.1 Triada de dovadadiff -> proof -> learning
Legenda
  • git diff = adevarul principal al evolutiei.
  • Extent = coloana de dovada pentru UI; XML confirma executia machine-readable.
  • Snapshot-urile .java sunt suport, nu adevar primar.
  • Daca lipseste una din cele trei coloane, raportarea va umple golul cu presupuneri.
2.4.2 Tipuri de artefacteadevar, invatare, guvernanta, handoff
Legenda
  • truth = dovezi primare, nu rezumate.
  • learning = orice MD care schimba rece urmatoarea miscare.
  • governance = ce face verdictul countable, legal si reluabil.
  • report handoff = ce permite raportarii sa publice fara reinterpretare.
  • Un artefact optional devine obligatoriu in momentul in care schimba rece urmatorul rerank sau verdictul final.

5. Regula diff-first pentru evolutiegit diff explica evolutia iterativa 
pe run
  runs/run_##/review/thin-no-index.diff = adevarul mic al claim-ului din run
  runs/run_##/review/source-delta-carrier.md = suport doar daca diff-ul singur nu e suficient
  SUMMARY.md = 1-3 linii despre mutatia istorica a run-ului

pe pachet
  review/root-source-thin-no-index.diff = adevarul mare compilat al pachetului
  raportul foloseste run diff pentru claim-ul mic si root diff pentru sinteza mare
Diff-ul este adevarul primar al evolutiei. Snapshot-urile de fisiere si explicatiile agentului sunt suport, nu substitut.
Formula canonica: run diff = mutatia mica a run-ului. root diff = mutatia compilata a pachetului. source-delta-carrier.md = suport doar daca diff-ul singur nu ajunge.
Pentru fiecare run publicat, SUMMARY.md trebuie sa spuna in 1-3 linii ce mutatie istorica a adus run-ul fata de starea anterioara a frontierei. Diff-ul ramane adevarul primar; sumarul doar il face redeschis rapid.
