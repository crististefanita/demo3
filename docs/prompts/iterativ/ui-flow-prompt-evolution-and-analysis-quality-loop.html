2) Mandatory Quality Review Of Analysis HTML

This section stays in the working meta prompt because the supervisor still owns local acceptance, local scoring thresholds, and local run-package decisions. The reusable orchestration baseline now also lives in prompt-evolution-orchestration-standard.html.
2.1 Review Scope ANALYSIS
  • MUST review the canonical analysis HTML generated by the discovery prompt.
  • MUST review every meaningful milestone preserved in the run-local analysis history.
  • MUST review the grouped package for the current run under RUN_ARCHIVE_ROOT/run_##, especially analysis/, review/, and manifests/.
  • MUST review whether the canonical artifact exposes a usable history index for the milestone family.
  • MUST review whether the per-run analysis package is coherent enough for an external audit without reading previous runs first.
  • MUST compare all completed run_##/analysis packages before promoting a best baseline.
  • MUST score the canonical artifact and the package as a whole.
  • MUST NOT score only the final HTML while ignoring whether the milestone history is coherent and useful.
2.2 Required Scoring Table QUALITY 
| Area | What was verified | Score | Risk / Note |
|---|---|---:|---|
The reusable scoring dimensions are standardized in the reporting and orchestration standards. Here, keep the local rule simple: score both the canonical analysis and the run package broadly enough to expose weaknesses in understanding, diagrams, history quality, package coherence, cross-run comparative strength, and explicit reporting-level overall quality.
The final analysis package is still below target if any one of these remains weak even when the overall score looks acceptable: business domain graph quality, actor state graph quality, business flow graph quality, or iteration-history quality.
2.3 Mandatory Interpretation QUALITY
  • MUST explain why the final score is below target when it is below target.
  • MUST identify whether the weakness came from discovery depth, HTML modeling, diagram quality, traceability, or prompt quality.
  • MUST identify whether the HTML family failed to explain the quality of the generated ATF package clearly enough.
  • MUST treat a low-quality analysis artifact as a signal that the execution prompt itself may need improvement.
  • MUST NOT treat the generated HTML as "good enough" merely because the tests passed.
2.4 Comparative Scoring Rule QUALITY
  • MUST compare the current analysis package against the immediately previous meta iteration when one exists.
  • MUST compare the full set of completed run folders, not only the newest run.
  • MUST decide explicitly which run currently represents the best execution-prompt baseline and why.
  • MUST NOT call the rerun "better" unless the score movement, weak-area improvement, or evidence package is materially stronger.
  • MUST NOT assume the latest run is automatically the best run.
  • MUST treat improved honesty and improved diagram readability as legitimate progress, even when raw coverage stayed similar.
2.5 Analysis Score As Test Quality Gate QUALITY
  • MUST treat the analysis score as a trust gate for the generated UI tests and shared-layer changes.
  • MUST NOT treat generated tests as sufficiently trustworthy when overall analysis quality is below QUALITY_TARGET_OVERALL.
  • MUST assume that weak business understanding, weak diagrams, or weak milestone history can invalidate apparent test quality even when the code compiles or the tests pass.
  • MUST rerun the process when the analysis package does not provide enough confidence to justify the generated tests.
  • MUST reject otherwise passing tests when flow modeling, actor-state modeling, or business diagrams are still too weak to explain what the tests truly validate.

3) Prompt Improvement Rule

3.1 When To Improve The Base Prompt PROMPT
  • MUST improve DISCOVERY_ATF_PROMPT_PATH when overall analysis quality is below QUALITY_TARGET_OVERALL.
  • MUST improve DISCOVERY_ATF_PROMPT_PATH when overall generated test-result quality is below QUALITY_TARGET_OVERALL.
  • MUST improve DISCOVERY_ATF_PROMPT_PATH when diagrams are weak, missing, or hard to read.
  • MUST improve DISCOVERY_ATF_PROMPT_PATH when milestone history is too thin, repetitive, or not useful as iteration evidence.
  • MUST improve DISCOVERY_ATF_PROMPT_PATH when run folders are not self-contained enough to support cross-run comparison.
  • MUST improve DISCOVERY_ATF_PROMPT_PATH when constants drift from the desired execution model, especially seed ownership, target package ownership, or HTML-family ownership.
  • MUST improve DISCOVERY_ATF_PROMPT_PATH when the final HTML is too framework-centered instead of business-centered.
  • MUST improve DISCOVERY_ATF_PROMPT_PATH when the generated ATF code is weak in layering, assertions, reuse, maintainability, or executable trustworthiness.
  • MUST NOT keep the prompt unchanged if the same analysis weaknesses are recurring.
3.2 What Can Be Improved PROMPT
  • business-first framing
  • analysis HTML section requirements
  • diagram requirements and readability constraints
  • run-folder grouping and per-run auditability
  • execution constants and repo-anchor clarity
  • milestone-history expectations
  • coverage-overlay rules
  • iteration evidence requirements
  • accessible-hyperlink exploration requirements
  • final scoring requirements
  • stop conditions and rerun conditions
3.3 Prompt Evolution Artifacts PROMPT
  • MUST save each improved prompt version inside the current run package under RUN_ARCHIVE_ROOT.
  • MUST save a short review/scoring HTML or markdown artifact for each meta iteration explaining what changed in the prompt and why.
  • MUST preserve prompt history even when ATF code is reverted.
  • MUST keep the active discovery prompt and any maintained companion format aligned when both exist.
  • MUST explain any intentional divergence from the strongest known previous prompt version.
  • MUST state explicitly when the HTML is the canonical prompt and the markdown file is only a concise companion.
  • MUST NOT lose the lesson trail that explains why prompt version N+1 is better than prompt version N.
  • MUST preserve promoted best-of references separately from raw run artifacts when a run produces especially strong prompt or analysis outputs.
3.4 Prompt Self-Scoring PROMPT

Each meta iteration may score the improved base prompt itself as a secondary diagnostic, but this must never replace scoring of the generated analysis output and generated ATF result package.

Minimum prompt scoring dimensions

  • diagram governance clarity
  • analysis HTML completeness guidance

3.5 Baseline Promotion Rule PROMPT

  • MUST keep the prior prompt version in history even when the new one becomes the baseline.
  • MUST NOT overwrite the lesson trail that explains why the new baseline won.
  • MUST fall back to the previous baseline if the rerun reveals weaker clarity, honesty, readability, or repo-fit.

4) Reset And Rerun Rule

4.1 Preserve Then Reset RESET
  1. preserve the canonical analysis HTML and run-local analysis history
  2. preserve every completed run folder under RUN_ARCHIVE_ROOT
  3. preserve prompt-evolution artifacts and prompt-review artifacts
  4. identify the ATF code/test/support changes introduced by the previous run
  5. revert only those ATF execution changes
  6. do not preserve superseded generated ATF code unless it is explicitly promoted as a kept baseline
  7. keep the improved base prompt
  8. rerun the UI-flow discovery process from a clean ATF code state
4.2 Reset Safety Rules RESET
  • MUST revert only the changes created by the run being superseded.
  • MUST preserve unrelated user changes and unrelated parallel work.
  • MUST preserve prompt files, analysis files, prompt-history files, and review artifacts.
  • MUST NOT carry forward superseded generated UI code just because it exists in the worktree.
  • MUST NOT use destructive reset logic that wipes the worktree indiscriminately.
  • MUST explain exactly what was reverted and why before the rerun begins.
4.3 Rerun Trigger RESET
  • MUST rerun the full UI-flow process after a meaningful prompt improvement, unless the remaining gap is clearly blocked by the product or by the environment.
  • MUST rerun when the analysis score is below target, even if the generated tests look superficially acceptable.
  • MUST rerun when the generated test-result quality is below target, even if the analysis package improved.
  • MUST rerun while the completed meta-run count is still below META_ITERATION_COUNT, even if the current run already looks strong.
  • MUST NOT stop at "prompt improved" without testing whether the improved prompt produces a better analysis artifact.
4.4 Keep And Reset Manifests RESET
  • MUST save a keep-manifest artifact for each meta iteration under the current run package.
  • MUST save a reset-manifest artifact for each meta iteration under the current run package.
  • MUST save run-grouped copies under the current run_## package, not only flat files in the archive root.
  • MUST list exactly which analysis files, prompt-history files, and review artifacts were preserved.
  • MUST list exactly which generated ATF files, support classes, and test files were reverted before the rerun.
  • MUST list any artifacts promoted into PROMOTED_BASELINE_ROOT and the reason they were promoted.
  • MUST NOT leave the reset decision implicit or reconstruct it later from memory.
4.5 Manifest Templates RESET 
keep-manifest
- preserved analysis canonical html
- preserved analysis milestone html files
- preserved current run-folder analysis package
- preserved prompt history files
- preserved prompt review files
- preserved files intentionally kept as baseline

reset-manifest
- reverted generated ui tests
- reverted generated ui support classes
- reverted generated helper/config changes tied to the superseded run
- untouched unrelated user changes
- untouched analysis and prompt-evolution artifacts
  • MUST keep these manifests short, explicit, and file-oriented.
  • MUST NOT replace them with vague prose like "reset previous changes".

5) Meta Iteration Loop

5.1 Mandatory Loop Contract QUALITY 
run base UI-flow prompt -> generate langgraph-business-understanding artifacts ->
score analysis artifacts -> score generated test-result package -> detect weaknesses -> improve base prompt ->
save prompt evolution artifacts -> group current run artifacts under docs/prompts/history/run_## -> preserve analysis artifacts ->
revert previous ATF code changes -> rerun from clean ATF state -> score again
This loop is complete only after the rerun proves whether the prompt improvement produced a better analysis artifact and a better generated test-result package.
The artifact that must keep getting better across reruns is langgraph-business-understanding.html plus its history, while the ATF code produced by weak runs is disposable unless explicitly promoted.
5.2 Quality Gates Before Rerun QUALITY
  • MUST finish the analysis scoring before deciding the next prompt improvement.
  • MUST finish the generated test-result scoring before deciding whether the current run is acceptable.
  • MUST finish the prompt self-scoring before promoting the new prompt as the rerun baseline.
  • MUST finish the current run package under RUN_ARCHIVE_ROOT/run_## before considering the iteration auditable.
  • MUST verify whether the execution prompt completed its own declared iteration depth and whether its canonical HTML met the execution prompt's own business-model and diagram expectations.
  • MUST finish the keep/reset manifests before the ATF state is cleaned for the next run.
  • MUST record whether the run improved breadth, determinism, truthfulness, prompt clarity, or only one of those dimensions.
  • MUST NOT rerun blindly without explaining what this next iteration is expected to improve.
5.3 Stop Conditions QUALITY
  • MUST continue until overall analysis quality >= QUALITY_TARGET_OVERALL and overall generated test-result quality >= QUALITY_TARGET_OVERALL, or a justified stop condition is reached.
  • MUST continue until META_ITERATION_COUNT run folders exist and are scored, unless an explicit documented block makes additional runs dishonest or impossible.
  • MUST continue when the execution prompt failed to complete its own required iteration depth, unless a real and documented block proved that more execution iterations were impossible or unsafe.
  • MUST continue when the generated canonical HTML is still weak in business domains, actor-state clarity, business-flow diagrams, or milestone usefulness, even if headline scores improved.
  • MUST continue when the analysis is still below target and there is an obvious safe prompt improvement left.
  • MUST continue when generated test-result quality is still below target and there is an obvious safe prompt improvement left.
  • MUST continue when obvious accessible-link frontier slices, distinct account routes, or meaningful CTA-driven flows were not yet attempted in the clean baseline environment.
  • MUST NOT treat a single strong run as completion when the configured loop depth has not yet been honored.
  • MAY stop only if the remaining weaknesses are blocked, unsafe, low-value, or clearly product-limited rather than prompt-limited.
  • MUST NOT stop merely because one rerun improved the score somewhat.
  • MUST NOT stop if the analysis score improved but the prompt self-score or reset discipline still regressed.
  • MUST NOT stop if the current base prompt still cannot repeatedly generate the expected reporting structure, diagrams, business traceability, and ATF code quality in the same run.
5.4 Cross-Run Baseline Selection QUALITY
  • MUST score every completed run_##/analysis package on the same 1-10 scale before choosing the best baseline.
  • MUST include an explicit cross-run comparison table and a named winner.
  • MUST keep the cross-run comparison table cumulative until all configured runs are complete.
  • MUST choose the best execution-prompt baseline through comparative quality, not chronology.
  • MUST allow a later run to win because it is more honest and more stable, even if it covers slightly fewer speculative claims than an earlier run.
  • MUST preserve the non-winning runs as historical evidence unless they are explicitly redundant and documented as such.
  • MUST NOT assume run_10 wins just because it is newest.

  • MUST NOT let the promoted best run differ from the run named in the final cross-run review artifact.
6.4 Core Principle CORE
The generated langgraph-business-understanding HTML family is a secondary output and a feedback layer about prompt quality. If the analysis artifact is still weak, or if the generated tests are still below the required quality level, improve the base prompt, keep the analysis evidence, undo only the superseded ATF execution changes, and rerun from a clean framework state until both the analysis quality and the generated test-result quality are strong enough or a real block is proven.
Promote known-good artifacts separately from raw run evidence. Keep the raw run folders immutable enough for audit, and copy the best analysis HTML, best prompt baselines, and best run summaries into PROMOTED_BASELINE_ROOT for future prompt evolution and comparison.
