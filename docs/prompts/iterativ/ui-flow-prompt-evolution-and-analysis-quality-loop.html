2.3 Winner Selection By Evidence QUALITY
  1. Compare the current run against the strongest kept run and the current promoted baseline.
  2. Name the decisive strength that makes the winner better.
  3. Name the decisive blocker that keeps each loser from promotion.
  4. Recommend promotion only if the winner clears QUALITY_TARGETS more honestly or more usefully than the incumbent, and publish an explicit no-winner / no-promotion verdict when the evidence does not support replacement.

Promotion gates

  • MUST NOT auto-promote the latest run.
  • MUST NOT promote a run whose comparative story cannot survive a later audit.
  • MUST classify targeted frontier passes as supporting proof when the full generated suite still fails. A targeted run may improve business understanding and guide the next patch, but it cannot become the promoted generated-output candidate until a clean retained full-suite rerun confirms the package.
  • MUST name whether the winning run won on business proof, reporting quality, orchestration discipline, execution robustness, or a combination, and avoid saying "better overall" without that split.
  • MUST NOT call a run a reliable winner if its package is too thin to reopen later without chat reconstruction.
  • MUST NOT call a run reporting-complete if its canonical ledger still has clipped command cells, horizontal overflow, or unreadable long-line rows that the reporting standard explicitly forbids.
  • MUST NOT present four identical reruns of the same narrow command as a sufficient full-family comparison unless the review explicitly proves that no broader symmetric retained-close evidence class was feasible.
  • MUST keep losing runs reviewable enough for later comparison.
  • MUST state one unambiguous Primary Retained Validation Run for the final verdict and explain why that run, rather than another preserved package, anchors trust.
  • MUST separate business-understanding winner, output / retained winner, and execution-robustness winner whenever they diverge. A cleaner retained package is not allowed to silently replace the stronger business-steering run.
  • MUST keep bootstrap-provenance wording short near the top of the report and move the detailed caveat into the audit sections. Long provenance paragraphs in the hero summary are a reporting-quality defect even when the underlying honesty is correct.
  • MUST say whether the winner is benchmark-grade, only a stronger local candidate, or merely a validated learning candidate. If the benchmark still leads, the final wording must not sound like unconditional graduation.
  • MUST penalize rounds that consume large budgets but cannot name their effective meaningful iterations or what each late checkpoint materially changed.
  • MUST treat copying into docs/goldenCopies/html-exemplars-to-follow as stricter than ordinary winner promotion. A round may win locally without earning exemplar status.
  • MUST allow golden-copy promotion only when the final canonical report beats every relevant local exemplar already stored there and no major reporting dimension regresses.
  • MUST NOT copy the current artifacts into the local exemplar library when the new report is merely prettier, merely different, or only stronger on one narrow frontier while a stored exemplar remains stronger overall as a reusable report.
  • MUST make the final golden-copy decision here, after the comparative review is closed. Discovery may produce candidate artifacts; only the quality-loop may authorize exemplar copying.
2.3.1 Comparative Close-Out Questions CORE fast final sanity check
  • MUST answer whether the round changed business direction or only improved retained-run clarity.
  • MUST answer whether the output winner is also the business-understanding winner, or only the strongest retained package.
  • MUST answer whether the strongest remaining gap is business depth, execution fragility, or packaging/provenance only.
  • MUST answer whether borrowed-family influence changed frontier depth or only reporting/orchestration quality.
  • MUST answer whether external bootstrap provenance changed only executability or also changed business depth. If it changed only executability, the review must keep that caveat out of the business-winner reasoning.
  • MUST answer whether the next move should deepen the same flow or pivot honestly to a new frontier.
  • MUST answer whether any seed-adjacent frontier or benchmark-significant branch remains materially open before allowing an early stop recommendation.
  • MUST answer whether the final report beat the local exemplar library or only beat the current round competitors.

Same-frontier reruns MUST be downgraded to packaging / orchestration gain only when they do not deepen business proof, reduce a real frontier gap, or materially strengthen retained evidence.

3) Prompt Evolution And Rerun Rules

3.1 Frontier Pressure And Prompt-Caused Gap Detection PROMPT
  • MUST evolve the discovery prompt when the weakness belongs to prompt wording, structure, missing guidance, weak boundaries, or insufficient evidence pressure.
  • MUST NOT call a rerun meaningful if a prompt-caused gap was diagnosed but the discovery prompt was not improved first.
  • MUST separate prompt-caused weakness from product limits, environment limits, and honest execution failure.
  • MUST keep prompt guidance concrete enough that the next run can be judged against it.
  • MUST treat a newly disproved frontier as a useful supervisory outcome, not just a failed attempt: the review must say what evidence broke the earlier hypothesis and why the next move stays, deepens, or pivots.
  • MUST treat benchmark comparison as a steering mechanism, not just a reporting appendix: if the comparison shows the current family only improved style or governance while frontier proof stayed flat, the next move must deepen proof or explicitly stop.
  • MUST ask whether a public or shell-level success has a visible continuation into a deeper protected or downstream state before declaring the frontier complete.
  • MUST treat visible downstream surfaces that fail materialization proof as a successful supervisory outcome when they prevent overclaiming and force a better frontier choice.
  • MUST keep route truth, protected-shell truth, flow truth, mechanism truth, and blocked truth distinct after login or other guarded transitions. If authenticated content is real while the route marker remains stale, delayed, or misleading, do not collapse shell/mechanism truth into fake route truth and do not discard the deeper proof just because the URL stayed noisy.
  • MUST flag frontier choice as weak when the round merely repeats a predecessor-displaced shell or route victory instead of deepening it materially or moving to a more important benchmark gap.
  • MUST flag frontier choice as weak when a round prefers another navigation-only increment even though a higher-value downstream business continuation was already reachable and observable in the live app.
  • MUST flag the supervising prompt as too permissive when it allows early stop without explicit closure of seed-adjacent frontiers, meaningful hyperlinks, and benchmark-significant gaps.
  • MUST ask whether the current round reduced a real business frontier gap or only reduced a comparison-honesty / package-traceability gap.
3.2 Preserve, Reset, Rerun RESET
  1. Preserve the full run package and review artifacts.
  2. Preserve prompt snapshots and comparative notes.
  3. Perform explicit pre-run cleanup of rerunnable generated output outside docs/prompts/iterativ, especially transient target/ artifacts, while keeping useful history and preserved run packages intact.
  4. Reset only superseded generated output under the governed execution layer.
  5. Keep the inherited reporting evidence and review history.
  6. Rerun the discovery prompt from a cleaner execution state.
  7. Re-score the new run against the previously kept winner.
Preserve learned business memory. Replace only execution output that the improved prompt is expected to regenerate more strongly.
Thin-package rule: if a kept run lacks a real review artifact, a real iteration ledger, or concrete command/result evidence, the next meta-iteration must repair package completeness before that run can anchor promotion.
  • MUST preserve analysis HTML, history, prompt snapshot, review, manifests, validation evidence, and run summary together for every kept run.
  • MUST reject or repair preserved run packages that cannot explain purpose, command shape, result, lesson, and promotion/context status from their own files.
  • MUST preserve every distinct real run that changed what the team knows: defect exposure, hardening, strategy shift, contract split, reporting lesson, or final validation.
  • MUST treat unused planned iteration depth as a defect unless the review proves that remaining frontiers were already blocked, unsafe, already exhausted against the benchmark, or honestly below value.
3.2.1 Fresh-Round Comparative Reset RESET history stays memory, not active winner
  • MUST reset active comparative state at the start of every new 2+2 or family-comparison round: no prior run keeps implicit winner status, retained status, or business-winner status just because it won a previous round.
  • MUST treat prior rounds as benchmark memory only unless the current round explicitly reopens them as named comparison anchors.
  • MUST rebuild the active scoreboard, winner split, frontier gap, and retained-run decision from the current round's own preserved runs.
  • MUST say explicitly which older runs were reopened as benchmark context and which older verdicts were intentionally reset out of active comparison scope.
  • MUST NOT let a prior winner, prior family verdict, or prior retained-close run survive into the new round as active evidence without fresh comparative reopening.
  • MUST reset proof-origin and retained-close labels for the new round and award them again only from evidence produced or explicitly reopened for that round.
  • MUST start every fresh dictated 2+2 or family-comparison round in a fresh round-scoped archive context with local numbering restarted at run_01. If older preserved runs remain on disk, move them out of the active round scope or mark them benchmark-only outside the active archive root before the new round begins.
  • MUST stop the round as invalid before counting any new meta-iteration if the task is validating the prompt family itself and the active archive scope still contains prior-round folders that were not explicitly reset out of scope.
  • MUST NOT excuse continued numbering such as run_55 by saying ranking reset semantically; for prompt-family validation rounds, archive-scope reset is part of the contract, not an optional implementation detail.
  • MUST reject the round as audit-invalid if the final user-facing verdict, canonical HTML, history HTML, and active run archive do not all describe the same concrete round. Style quality or green tests cannot compensate for message-to-artifact mismatch.
  • MUST reject the page as reporting-incomplete if validation ledgers or artifact-reference tables still violate the reporting standard's long-line containment rules after the final render-verification pass.
Historical memory is allowed. Historical carry-over verdicts are not. A new round starts with an empty active ranking unless the review explicitly reopens older anchors as benchmark evidence.
3.3 Honesty And Failure Rules RESET
  • MUST NOT invent evidence that is not visible in the run package.
  • MUST NOT confuse partial, blocked, failed, and validated states.
  • MUST NOT claim a clean rerun when reset safety forced a partial reset.
  • MUST NOT treat prompt edits as success until another governed run proves a stronger result.
  • MUST NOT stop at prompt commentary, HTML cleanup, or standards editing when the local task is to execute the quality-loop.
  • MUST NOT publish ad hoc execution notes, path-contract checks, or temporary review scratch files inside docs/prompts/iterativ. That directory is reserved for governed prompt artifacts.
  • MUST NOT treat public-support success, shell success, or navigation success alone as frontier closure when the application already exposed a deeper protected or downstream continuation.
  • MUST downgrade the claim when the rerun is partial, blocked, or below target.
  • MUST name when a shared-engine bootstrap or precursor defect remains unresolved and the round advances only through an explicit local bootstrap or workaround.
  • MUST say explicitly when borrowed-family influence changed only form, provenance, or packaging, and not frontier depth.
  • MUST publish bootstrap-state, missing-input-state, no-incumbent-baseline-state, or single-candidate-state explicitly when those are the honest execution outcomes.
