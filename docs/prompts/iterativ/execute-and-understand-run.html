2.2 Split runtime -> orchestrator -> executie -> raportare cine fixeaza, cine decide, cine publica
Legenda
  • Runtime-ul fixeaza configuratia comuna; nu decide singur nici legalitatea, nici publicarea.
  • Orchestratorul decide daca urmatorul slot este legal.
  • Executia produce artefactele, explica progresul si fixarea rece a pachetului.
  • Raportarea doar publica adevarul deja fixat.
2.3 Ciclul unui singur run ce trebuie sa faca fiecare run
Ce trebuie sa ramana in minte
  • Run-ul trebuie sa lase dovada redeschisibila: diff, proof, state si lectie.
  • Ce nu este salvat pe disk nu poate fi folosit sigur de urmatorul agent.
  • Fara diff, proof si stare de run, fixarea rece nu este completa.
  • Un run explica targetul activ sau publica defect / blocker sever.
  • Invatarea decisiva trebuie salvata pe disk, nu lasata in chat.

2.4 Regula de grupare in acelasi runcand o zona merita mai multe teste
Daca ai ajuns aiciJudeciForma buna
pagina / stare / functionalitate cu comportamente business distincte poate aceeasi ancora inchide targetul activ fara sweep mecanic? grupezi testele inrudite in acelasi run
ancora care nu poate sustine targetul activ blocker real sau frontiera aleasa gresit? publici defect / blocker sever si rerank
zona cu alta identitate business sau alta dovada separarea ajuta claritatea, izolarea sau rerank-ul? desparti doar daca noul run poate inchide targetul activ
Legenda
  • Exemplu bun: aceeasi zona favorites poate produce in acelasi run empty-state, persistence si collection behavior, daca toate vin din aceeasi ancora functionala.
  • Exemplu prost: cinci variante apropiate ale aceluiasi click sau ale aceluiasi blocker doar ca sa umfli numarul.
  • Gruparea buna se face pe functionalitate si adevar business, nu pe comoditatea implementarii.
  • La finalul fiecarui run: spune daca targetul activ a fost inchis.
  • Daca nu: verdictul este defect sau blocker sever.
2.4.1 Detalii de executieinainte / in timpul / dupa run
MomentChecklist receNu accepta

Dupa salveaza diff, proof, XML/Extent, state, feedback si accounting rece lectii ramase doar in chat sau dovada fara link redeschisibil

4. Taxonomia artefactelormereu / conditionat / suport
  • Mandatory Artifact Spine = contractul.
  • Taxonomia = severitatea artefactelor.
  • Nu muta ownership-ul.
NivelIntra aiciRegula rece
mereudiff relevant, proof relevant, state files, close-outfara ele, run-ul sau pachetul nu este fixat complet
conditionatsource-delta-carrier.md, blocker/repair/frontier MD, subsectiune specializatadevin obligatorii cand schimba claim-ul, interpretarea sau rerank-ul
suportnote scurte, explicatii locale, context auxiliarse pastreaza doar daca ajuta redeschiderea rece
  • Nu salva artefacte ca decor.
  • Pastreaza doar adevar, interpretare necesara sau suport real.
  • Daca nu ajuta urmatorul agent, nu intra in fixarea rece.
5. Regula diff-first pentru evolutie PROOFgit diff explica evolutia iterativa
DacaSalveziRol
un singur run a schimbat claim-ul publicatruns/run_##/review/thin-no-index.diffmutatia mica a run-ului
mai multe run-uri au schimbat pachetulreview/root-source-thin-no-index.diffmutatia compilata a pachetului
diff-ul singur nu explica clar claim-ulsource-delta-carrier.mdsuport de interpretare, nu adevar primar
pe run
  runs/run_##/review/thin-no-index.diff = adevarul mic al claim-ului din run
  runs/run_##/review/source-delta-carrier.md = suport doar daca diff-ul nu ajunge
  SUMMARY.md = 1-3 linii despre mutatia istorica a run-ului

pe pachet
  review/root-source-thin-no-index.diff = mutatia compilata a pachetului
  raportul foloseste run diff pentru claim-ul mic si root diff pentru sinteza mare
  • Diff-ul este adevarul primar al evolutiei.
  • Snapshot-urile de fisiere sunt suport.
  • Explicatiile agentului sunt suport.
  • source-delta-carrier.md explica selectia sau sensul diff-ului.
  • Daca diff-ul lipseste, carrier-ul nu salveaza claim-ul.
  • SUMMARY.md: 1-3 linii per run.
  • Spune mutatia istorica fata de frontiera anterioara.
  • Diff-ul ramane adevar primar.
  • Sumarul doar il face redeschis rapid.
