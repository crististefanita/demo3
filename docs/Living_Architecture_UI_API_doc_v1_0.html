2) Architecture diagrams CORE

2.1 Module structure (static) Component
Explanation Structure and responsibilities

What it illustrates:

  • Separation of responsibilities between the UI, API, and Reporting components, with tests strictly limited to flow orchestration.
  • UI: interactions, waits, and validations are isolated from tests and grouped into dedicated layers.
  • API: business flows are orchestrated through API Steps, which separate HTTP calls and assertions from the tests themselves.
  • Reporting: the test lifecycle and step semantics are centralized and independent from the UI/API engines.

Why it matters:

  • Allows UI, API, or reporting engines to be changed without impacting tests.
  • Reduces coupling and logic duplication by introducing a dedicated semantic orchestration layer.
  • Provides a consistent and predictable model for testing and reporting.

3) Canonical rules — MUST / MUST NOT CORE

3.1 Configuration & reporting integration CORE Contract-driven

Config

  • MUST all framework settings be defined in framework.properties.
  • MUST configuration access happen exclusively through ConfigManager.
  • MUST there be a single ConfigManager, located in the core module and used by all modules (UI and API).
  • MUST ConfigManager provide access for reading configuration values through require(key) and get(key, default).

Reporting integration

  • MUST the framework interact with reporting exclusively through the canonical contract defined in Section 3.5.
  • MUST NOT execution layers (UI, API, Validation) depend on engines, adapters, or internal reporting mechanisms.
3.2 UI: interactions and waits UI UIActions + UIEngine + WaitUtils

UI interactions

  • MUST all UI interactions be executed exclusively through UIActions.
  • MUST UIEngine expose only basic operations (click, type, getText, wait), without test logic or reporting.
  • MUST the framework provide two concrete implementations of UIEngine (Selenium and Playwright), selectable through configuration.
  • MUST the active UIEngine standardize browser size for execution and screenshots (default 2560x1440), regardless of engine.

Waits

  • MUST all explicit waits be implemented exclusively through WaitUtils.
  • MUST the decision to apply an explicit wait be made inside UIActions, in an engine-aware manner.
  • MUST NOT WaitUtils emit FAIL directly; FAIL is controlled exclusively at the step level.

Engine-dependent wait strategy

  • MUST UIEngine expose the method boolean supportsAutoWait().
  • MUST Selenium-based engines return false, while Playwright-based engines return true.
  • MUST UIActions decide explicit-wait application before interactions.
private static void waitIfRequired(String cssSelector) {
  if (!DriverManager.getEngine().supportsAutoWait()) {
    WaitUtils.waitForVisible(cssSelector);
  }
}
3.3 Validation Layer UI Assertions & verdict
  • MUST all assertions be implemented exclusively in the Validation Layer.
  • MUST validations be exposed through reusable classes.
  • MAY the Validation Layer create Validate steps in the report using StepLogger.
  • MUST a validation failure mark the test as FAILED without stopping execution of subsequent tests.
3.4 API: services API Steps → Services → Validation

Test Layer

  • MUST tests orchestrate business flows exclusively through API Steps, Factories / Builders, and Validators.
  • MUST NOT tests execute HTTP calls or perform reporting; any direct use of reporting mechanisms from the Test Layer is forbidden.

API Steps

  • MUST API Steps orchestrate semantic business flows.
  • MUST API Steps call Services to execute HTTP calls.
  • MUST API Steps call the Validation Layer for assertions.
  • MUST NOT API Steps execute HTTP calls directly or construct payloads.

Services

  • MUST Services define the endpoint through basePath.
  • MUST Services expose business methods.
  • MUST NOT Services contain assertions or validations.

HTTP Execution

  • MUST HTTP-call execution be delegated to a reusable shared layer.
  • MUST the full URL be built as {base.url.api} + relativePath.
  • MUST the execution layer return the Response object.

Request / Response Models

  • MUST request models be constructed exclusively through Factories or Builders.
  • MUST NOT tests or API Steps manually construct JSON payloads.

Reporting for API

  • MUST every HTTP call be wrapped in an active semantic step.
  • MUST the semantic call details (method, path, URL, status) be logged within the active step.
  • MUST HTTP payloads (request and response) be logged exclusively through the canonical reporting contract.
  • MUST HTTP payloads be formatted as JSON (pretty-printed) before logging.

Semantics and naming for API steps

  • MUST HTTP call steps use the format {HTTP METHOD} {relativePath} (e.g. POST /users); the full URL MUST be included in details.
  • MUST validation steps express expectations in a declarative format Validate: {expectations} (ex. status=201, body~=userId, (JSON) schema...).
  • MAY validation details (actual values, additional messages) be logged in the step content.
3.5 Canonical reporting contractCORE StepLogger

Principle

  • MUST the framework expose a single canonical mechanism for writing reporting information.
  • MUST this mechanism be accessible to all execution layers (UIActions, Services, Validation).
  • MUST execution layers emit reporting information exclusively through StepLogger, without depending on engines or concrete adapters.

Canonical API - required methods

  • MUST there be a method for starting a semantic step: StepLogger.startStep(String title).
  • MUST there be a method for logging textual details within an active step: StepLogger.logDetail(String detail).
  • MUST there be a method for marking a step as successful: StepLogger.pass(String message).
  • MUST there be a method for marking a step as failed: StepLogger.fail(String message).
  • MUST there be a method for attaching structured content as a code block: StepLogger.logCodeBlock(String content).

Usage rules

  • MUST each semantic step be created explicitly through StepLogger.startStep(...).
  • MUST a step be finalized exactly once through pass(...) or fail(...).
  • MUST NOT allow a new step to start while an active step already exists.
  • MUST NOT call logDetail(...) or logCodeBlock(...) outside an active step.

Limitations

  • MUST NOT execution layers (UIActions, Services, Validation) control test start or test end.
3.6 Global constraintsCORE
  • MUST all framework dependencies be available at runtime.
  • MUST NOT dependencies be declared with <scope>test</scope>.
  • MUST NOT the method names wait, notify, notifyAll, finalize be used.
  • MUST there be a single pom.xml at the project root.
  • MUST NOT separate Maven modules (<modules>) be defined.

4) Reporting (engines, contract, and internal details) REPORTING

This section groups all internal reporting details (engines Extent | Allure, adapters, diagrams).

4.1 Reporting — execution flows & lifecycle Flows, lifecycle, contract
4.1.1 UI Flow -> Reporting Sequence

The flow shows how UI actions become report steps and how they are routed to the reporting engine through abstractions.

Explanation transforming UI actions into reported steps
  • What it illustrates: The transformation of a UI action into a reported semantic step.
  • Tests: Trigger high-level actions independent of the UI engine in use (Selenium, Playwright).
  • UIActions: Decide whether a wait is required and execute the action through the engine.
  • StepLogger: Creates the step, logs details, and controls PASS/FAIL.
  • ReportingManager: Selects the reporting adapter (Extent / Allure).
  • Key rule: Each step can have only one FAIL; the engine and tests do not log directly.
4.1.2 API Flow -> Reporting Sequence

Symmetric flow: HTTP calls and validations become report steps through the same contract (StepLogger -> ReportingManager -> ReportLogger).

Explanation unified reporting for API
  • What it illustrates: The transformation of HTTP calls and validations into reported steps.
  • Services: Execute HTTP calls through ApiClient.
  • Validation: Run assertions exclusively inside logged steps.
  • StepLogger: Ensures consistent logging semantics.
  • Benefit: All tests use the same reporting model.
4.1.3 TestNG + StepLogger flow (failure rules) Sequence
Explanation Test lifecycle and failure rules
  • What it illustrates: Separation of responsibilities between the test lifecycle and the semantics of reported steps.
  • Separation of responsibilities:
    • TestListener: manages test start and end (TestNG events) and creates/closes the reporting context at the test level.
    • StepLogger: defines the semantic step and controls correct reporting, being the only component allowed to emit FAIL inside a step.
  • Implicit diagram rules:
    • MUST a step be able to have only one FAIL.
    • MUST details be logged exclusively inside an active step.
    • MUST any exception occurring outside steps produce FAIL at the test level.
    • MUST NOT allow StepLogger.logDetail(...) calls outside an active step.
4.1.4 Lifecycle TestNG + reporting (UI) Sequence
Explanation screenshot and UI failure
  • What it illustrates: execution of a UI test with the reused reporting mechanism.
  • Steps in UI tests:
    • UIActions execute UI interactions and create semantic steps.
    • Details about locator, value, and wait are logged in the step.
    • Screenshot is attached only once on FAIL, at the test level.
4.1.5 Lifecycle TestNG + reporting (API) Sequence
Explanation API with shared reporting
  • What it illustrates: execution of an API test with the reused reporting mechanism.
  • Steps in API tests:
    • Services execute HTTP requests and create HTTP ... steps with the method, path, URL, and relevant parameters.
    • Payloads (request/response) are attached as code blocks in the report.
    • Validations run assertions in separate steps, distinct from HTTP-call steps.
    • MUST HTTP calls be reported in separate steps.
    • MUST validations be reported in steps different from HTTP calls.
4.1.6 Node structure in the report UX model
Explanation report contract
  • What it illustrates: defines the report “contract”: where each category of information is placed so the report stays scannable and free of duplication.
    • Test Node contains the test name, description, and final status.
    • Test Node does not contain stacktrace or payloads.
    • Step Node contains semantic title, details, payloads, and the step result.
    • Stacktrace appears only once, exclusively in the step that failed.
    Minimum recommended detail set (standardization):

    UI step — details

    • locator=cssSelector
    • value=... (for type/select)
    • wait=visible/clickable (if applied explicitly)

    API step — details

    • method=POST, path=/public/v2/users
    • url=https://.../public/v2/users
    • status=201 (after receiving the response)
Rule: stacktrace appears only once, only in the step that failed. The test node does not receive stacktrace.
4.2 Reporting: engines & adapters Extent / Allure
  • MUST reporting-engine selection be controlled exclusively by ReportingManager, based on the reporting.engine property.
  • MUST reporting adapters implement the ReportLogger interface.
  • MUST reporting adapters be accessed exclusively through ReportingManager.
  • MUST NOT tests, services, or validations call reporting adapters directly.
  • MUST inactive adapters be NO-OP and not throw exceptions.
Engine-specific rules:
  • Extent: the startTest, endTest, and flush methods are used for explicit control of the test lifecycle.
  • Allure: the test lifecycle is controlled by AllureTestNg, while lifecycle methods from ReportLogger are treated as NO-OP.
  • Allure: listener order is mandatory (AllureTestNg before TestListener) for correct artifact attachment and test closure.
4.3 Logging & step semantics StepLogger + TestListener
The rules in this section (4.3 Logging & step semantics) apply exclusively to execution layers (UIActions, Services, Validation). Test Layer is not a valid consumer of StepLogger.

Applying the canonical rules for steps

  • MUST each semantic step be explicitly started through StepLogger.startStep(...).
  • MUST StepLogger be the only mechanism allowed to mark a step as FAIL.
  • MUST each step have exactly one final result (PASS or FAIL).
  • MUST NOT allow a new step to start while an active step already exists.

Details and exceptions

  • MUST steps be created only in the context of an active test.
  • MUST calls to logDetail(...), pass(...), and fail(...) occur exclusively inside an active step.
  • MUST any exception occurring outside an active step produce a test-level FAIL, handled through TestListener.

Stacktrace and screenshot

  • MUST stacktrace be reported only once, exclusively in the step that failed, as a code block.
  • MUST screenshot be captured exclusively from TestListener, only once per FAIL test, at the test level.

Console logging

  • MUST console logging be performed exclusively through StepLogger.
  • MUST each step be displayed with a semantic title on a separate line.
  • MUST step details be displayed indented, one item per line.
  • MUST the step result be displayed on the last line, with visual status (PASS / FAIL).
  • MAY details be omitted when console.details.enabled=false.

4.4 Reporting: extensibility & operations Engines & maintenance

Extensibility

  • MUST any reporting engine be integrated exclusively through an adapter that implements ReportLogger.
  • MUST engine selection and instantiation be controlled exclusively by ReportingManager.
  • MUST NOT require changes in tests or execution layers to add a new engine.

Fail-safety

  • MUST unavailable or invalid engines degrade safely through NO-OP behavior.
  • MUST reporting not interrupt test execution.

Operations

  • MUST external-engine listener order be respected according to engine requirements.
  • MUST NOT tests or execution layers interact directly with engines or adapters.

Objective

  • MUST the reporting architecture allow engines to be changed without impacting tests.

Rules: Listeners & BaseTestUI UI
  • MUST there be a canonical base class named BaseTestUI.
  • MUST all test classes (UI) extend BaseTestUI.
  • MUST BaseTestUI declare listeners through @Listeners, in the following order: AllureTestNg, TestListener.
  • MUST shared infrastructure initialization happen once per class through @BeforeClass(alwaysRun = true).
@Listeners({
    io.qameta.allure.testng.AllureTestNg.class,
    com.endava.ai.core.listener.TestListener.class
})
public abstract class BaseTestUI {

  @BeforeClass(alwaysRun = true)
  public void setUp() {
    DriverManager.initEngine();
  }

  @AfterClass(alwaysRun = true)
  public void tearDown() {
    DriverManager.quitEngine();
  }
}
Rules: API tests API
  • Target API (config): {base.url.api} = https://gorest.co.in
  • MUST tests be fully independent (order irrelevant)
  • MUST clean up created data
  • MUST for GET / UPDATE / DELETE, each test create its entity in setup
  • MUST hierarchy: Positive / Negative / End-to-end
  • MUST E2E: Create -> Get -> Update -> Delete -> Verify deletion
  • MUST JSON Schema validation inside logged steps

Rules: Listeners & BaseTestAPI API
  • MUST there be a canonical base class named BaseTestAPI.
  • MUST all test classes (API) extend BaseTestAPI.
  • MUST BaseTestAPI declare listeners through @Listeners, in the following order: AllureTestNg, TestListener.
  • MUST shared infrastructure initialization happen once per class through @BeforeClass(alwaysRun = true).
@Listeners({
    io.qameta.allure.testng.AllureTestNg.class,
    com.endava.ai.core.listener.TestListener.class
})
public abstract class BaseTestAPI {

  @BeforeClass(alwaysRun = true)
  public void baseSetup() {
    ApiClient.init();
    requireTokenForWrites();
  }

  protected void requireTokenForWrites() {
    String token = ConfigManager.require("auth.token");
    if (token.isBlank())
      throw new SkipException("auth.token is not set; GoRest write operations require a token");
  }
}
